## 一、引言

### 1.1 项目背景

随着我国老龄化进程加速，65岁以上人口已超过2亿，其中大量为独居、慢病群体。这些老年人在日常健康管理中面临诸多困难：用药记录靠纸笔容易遗忘、健康数据分散难以追踪、子女远程照护缺乏信息通道、紧急情况求助不便。与此同时，AI大模型技术的快速发展为解决这些问题提供了新的可能性。

然而，直接使用大模型对话框（如ChatGPT、通义千问等）存在根本性局限：每次对话从零开始、无法持久记忆用户信息、不能执行实际操作（如存储数据、设置提醒）、不能主动服务（如定时提醒吃药）、数据在云端存在隐私风险。正如课程中所述——**对话框是"问"，应用是"做"**。用户愿意多一步选择独立应用，只因为它能帮他"做"事，而不仅仅帮他"说"话。

### 1.2 项目目标

本项目旨在开发一个面向65岁以上中老年人（独居、慢病群体）及其子女（照护者）的AI健康管理助手——**银发健康助手**。核心目标包括：

1. **用最简单的对话帮助记录用药与健康数据**：用户只需说"我今天散步了30分钟"，系统自动识别意图、提取信息、存储数据
2. **提供安全健康科普咨询**：在医疗安全底线之上，基于用户画像提供个性化健康建议
3. **支持子女在授权下查看父母数据**：通过家庭绑定机制实现远程照护信息通道
4. **银发适老化设计**：大字体、高对比度、语音交互，降低使用门槛

### 1.3 核心设计思想

本应用的设计遵循"超越对话框"的六大核心要素：

| 要素 | 本应用实现 | 对话框无法提供 |
|------|-----------|---------------|
| **执行能力** | 自动存储健康数据、设置提醒、异常告警 | 只能生成文本建议 |
| **持久记忆** | ChromaDB语义检索 + JSON文件持久化 + 用户画像 | 每次对话从零开始 |
| **场景深度** | 慢病知识库、禁忌运动表、饮食禁忌映射 | 通用但不专业 |
| **持续运行** | 定时提醒引擎、30秒轮询、浏览器通知 | 被动响应，不会主动提醒 |
| **工具集成** | 天气API、语音识别/播报、子女端数据共享 | 孤立无连接 |
| **智能理解** | 三级意图识别架构，自然语言即用 | 需要精心写prompt |

## 二、需求分析

### 2.1 目标用户

本应用面向两类核心用户：

**主要用户：65岁以上中老年人**
- 特征：独居或与子女分居、患有慢性病（高血压/糖尿病/骨质疏松等）、智能手机使用能力有限
- 核心需求：简单记录健康数据、按时用药提醒、紧急求助、健康咨询

**辅助用户：子女（照护者）**
- 特征：工作繁忙、与父母异地、担心父母健康
- 核心需求：远程查看父母健康数据、用药依从性监控、异常告警通知

## 四、核心模块详细设计与实现

### 4.1 三级意图识别架构

意图识别是本系统的核心，采用分层处理架构，从简单到复杂逐级处理：

```
┌─────────────────────────────────────────────────────┐

        │  第一层：快速匹配（零成本）│  <5毫秒

        │  (firstLayer.js         │
        │   + fastReplies.js)     │
        └───────────┬─────────────┘
                    │
         ┌──────────┴──────────┐
         │ 匹配到？              │
         ▼                      ▼
        是                     否
        │                      │
        ▼                      ▼

                         └───────────┬─────────────┘
                                     │
                                     ▼

                          │  识别结果            │
                          │  - 意图类型          │
                          │  - 置信度            │

                                     │
                                     ▼
                          ┌─────────────────────┐
                          │  第三层：安全检查    │
                          │  代码强制执行规则    │
                          │  (safetyRules.js)    │
                          └───────────┬───────────┘
                                     │
                          ┌──────────┴──────────┐
                          │  安全？              │
                          ▼                      ▼
                        是                     否

```

#### 4.1.1 第一层：正则词库快速匹配

**设计目标**：用正则匹配简单、无歧义的消息，不调用大模型，零成本。

**实现文件**：`frontend/intent/firstLayer.js` + `frontend/intent/fastReplies.js`

`firstLayer.js` 定义8大词库类型，约60个关键词；`fastReplies.js` 定义寒暄/确认/告别等快速直出模式。

| 词库类型 | 示例关键词 | 返回行为 |
|---------|-----------|---------|
| 寒暄问候 | 你好、您好、早上好、晚安 | 直接返回问候语 |
| 感谢确认 | 谢谢、好的、可以、没问题 | 直接返回确认语 |
| 告别结束 | 再见、拜拜、明天见 | 直接返回告别语 |
| 使用帮助 | 帮助、怎么用、你能做什么 | 返回功能介绍 |
| 用药提醒开关 | 提醒我吃药、关闭提醒 | 路由到提醒模块 |
| 强格式记录 | 血压130/80、血糖5.6 | 直接提取数值存储 |
| 子女查看数据 | 我妈的血压、爸爸的用药 | 路由到子女查看模块 |
| 情绪关怀 | 心情不好、有点累 | 路由到情绪关怀模块 |

**匹配条件**：消息长度≤18字、无歧义、不需要后续参数提取。

**核心代码逻辑**（`firstLayer.js`）：

```javascript
export function runFirstLayer(text, profile) {
  // 长度检查：超过18字通常不是简单对话
  if (text.length > 18) return null;
  
  // 依次检查8大词库
  for (const rule of RULES) {
    if (rule.exact && text === rule.keyword) return rule.result;
    if (rule.pattern && rule.pattern.test(text)) return rule.result;
    if (rule.includes && text.includes(rule.keyword)) return rule.result;
  }
  return null; // 未匹配，交给第二层
}
```

**设计决策**：宁可漏也不要错——漏了交给第二层AI处理，错了就出问题。

#### 4.1.2 第二层：AI意图识别与参数抽取

**设计目标**：对于复杂消息，调用大模型判断意图并提取关键信息。

**实现文件**：`frontend/app.js` 中的 `callLLM()` 函数（第383行起）

- **大模型**：阿里云通义千问 `qwen-plus`
- **API端点**：`https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`（OpenAI兼容模式）
- **temperature**：0.3（低温度保证稳定输出）

**提示词设计**（`frontend/intent/secondLayerPrompt.md`）：

提示词包含以下七个部分：

1. **角色定义**：告知模型"你是银发健康助手的意图识别模块"
2. **意图列表**：6大类16个子意图，用表格清晰列出
3. **用户画像**：年龄、性别、慢性病史、过敏史、当前用药
4. **会话记忆**：最近3条对话历史
5. **用户消息**：用「」包裹原始消息
6. **任务说明**：判断意图、提取信息、给出置信度
7. **输出格式**：严格JSON（intent, confidence, reasoning, extractedData, needClarify）

**意图分类体系**（本系统实际支持的意图）：

| 大类 | 代码 | 子意图数 | 核心能力 |
|------|------|---------|---------|

#### 4.1.3 第三层：安全检查

**设计目标**：不管AI返回什么，代码强制执行安全规则。

**核心观点**：第二层AI识别 ≠ 第三层安全检查。AI可能漏检、误判，而健康领域容不得错误。

**实现文件**：`frontend/intent/safetyRules.js`，`runSafetyCheck()` 函数

四级安全体系：

| 级别 | 行为 | 适用场景 | 示例 |
|------|------|---------|------|
| **BLOCK** | 拦截，不执行 | 自伤/自杀、危险用药、胸痛/呼吸困难 | 高血压患者推荐快跑 |
| **URGENT** | 允许执行+紧急提示 | 需要立即就医的症状 | 血压≥180/120 |
| **WARN** | 允许执行+警告提示 | 潜在风险行为 | 饮酒后运动、擅自调药 |
| **PASS** | 正常执行 | 普通记录/查询/科普 | 记录今天散步30分钟 |

**安全规则实现**（`safetyRules.js` 核心逻辑）：

绝对禁止规则：
- 自伤/自杀相关关键词 → BLOCK
- 违法/危险用药 → BLOCK
- 诈骗/敏感信息泄露 → BLOCK

条件警告规则：
- 擅自调整用药 → WARN
- 高危症状待确认（胸痛/呼吸困难/中风表现/意识障碍） → URGENT
- 饮酒后/空腹运动风险 → WARN
- 高强度运动风险 → WARN

健康顾问安全集成（`healthAdvisor.js`）：
- 慢性疾病饮食禁忌映射（高血压/糖尿病/高血脂/冠心病）
- 根据用户慢性病自动过滤不安全的饮食建议

### 4.2 记忆系统

```

    │

```

#### 4.9.1 设计原则

| 原则 | 实现措施 |
|------|---------|
| **看得清** | 4档字体大小（基准24px-30px）、高对比度配色、大模块间距 |
| **点得着** | 大按钮（最小48px×48px触摸目标）、简化导航（底部4Tab）、减少操作层级 |
| **听得懂** | 语音播报关键信息、简洁友好的文案、避免技术术语 |

### 7.1 实践总结

本项目围绕"超越对话框"的核心设计思想，设计并实现了一个面向65+中老年人的AI健康管理助手——银发健康助手。通过实践，我们深刻理解了以下要点：

#### 7.1.1 对"超越对话框"六大要素的实践验证

| 要素 | 实践验证 | 代码实现位置 |
|------|---------|-------------|
| **执行能力** | 用户说"我今天散步了30分钟"，系统自动识别意图、提取信息、存储数据 | `app.js:saveToMemory()` + `dataExtractor.js` + 后端`memory_service.py:save_record()` |
| **持久记忆** | ChromaDB语义检索 + JSON文件持久化，用户画像跨会话保持 | `memory_service.py:semantic_search()` + `memoryCache.js` + 对话历史持久化 |
| **场景深度** | 慢病禁忌运动表、饮食禁忌映射、天气健康建议 | `safetyRules.js` + `healthAdvisor.js` + `weather_service.py` |
| **持续运行** | 30秒轮询提醒、10分钟重提醒、漏服通知子女 | `reminderService.js` + `reminderServiceEnhanced.js` + `reminder_engine.py` |
| **工具集成** | 天气API、语音识别/播报、子女端数据共享 | `weather_service.py` + `voiceService.js` + `ttsService.js` + `family_service.py` |
| **智能理解** | 三级意图识别，自然语言即用，无需学习操作 | `firstLayer.js` + `app.js:callLLM()` + `safetyRules.js` |

#### 7.1.2 三级意图识别架构的实践价值

1. **成本优化**：第一层正则匹配处理约40%的简单消息，零API调用成本
2. **安全可靠**：第三层代码安全检查独立于AI判断，100%执行安全规则
3. **降级容错**：LLM调用失败时自动降级到正则匹配，系统不会因外部服务故障而不可用
4. **置信度机制**：不确定时降级处理，避免误操作

#### 7.1.3 银发适老化设计的实践心得

1. **字体大小可调**是刚需而非可选项，老年人视力差异大
2. **语音交互**是降低使用门槛的关键，语音识别+播报让不擅长打字的用户也能使用
3. **简化导航**比美化界面更重要，4Tab底部导航比复杂的侧边栏更友好
4. **主动服务**对老年用户尤其重要，他们不会主动去"查看"，需要系统主动"提醒"

### 7.2 不足与展望

#### 7.2.1 当前不足

1. **API密钥安全**：通义千问API密钥硬编码在前端`app.js`中，生产环境需要移至后端代理
2. **并发性能**：Flask默认单线程模式，多用户并发时性能受限
3. **离线能力**：依赖后端服务和LLM API，网络断开时功能受限
4. **测试覆盖**：仅有`profileExtractor.test.js`和`memoryTest.js`，单元测试覆盖不够全面
5. **数据迁移**：JSON文件存储不便于数据迁移和备份

#### 7.2.2 改进方向

1. **API密钥安全**：将LLM调用移至后端代理，前端不暴露API密钥
2. **数据库升级**：将JSON文件存储迁移至SQLite/MySQL，提升查询性能和数据安全
3. **PWA离线支持**：使用Service Worker缓存关键资源，支持离线使用
4. **多语言支持**：支持方言识别（粤语、四川话等），覆盖更多老年用户
5. **智能体扩展**：引入多Agent架构，不同健康领域由专业Agent处理

### 7.3 核心结论

> **对话框是"问"，应用是"做"。**
> 
> 银发健康助手的核心价值不在于"能对话"，而在于"能做事"——能记住用户的健康数据、能主动提醒用药、能在危险时拦截、能让子女远程照护。这些是任何大模型对话框都无法提供的价值。
>
> 不是做一个"更好的对话框"，而是做一个"真正的智能伙伴"。
