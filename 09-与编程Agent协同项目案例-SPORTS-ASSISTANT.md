## 一、课程目标

### 你将学会

| 能力          | 说明                    |
| ----------- | --------------------- |
| **理解Agent** | 知道编程Agent能做什么、不能做什么   |
| **有效沟通**    | 学会用Agent能理解的方式表达需求    |
| **分步推进**    | 掌握将大任务拆解为Agent可执行的小步骤 |
| **质量把控**    | 学会验证Agent的输出，确保代码质量   |
| **上下文管理**   | 学会维护项目上下文，让Agent持续高效  |

### 核心认知

> **Agent不是替代你编程，而是与你协同编程。**
> 
> 你是架构师和决策者，Agent是执行者和实现者。
> 
> 协同的质量，取决于你给出指令的质量。

## 二、认识编程Agent

### 2.1 Agent是什么

编程Agent（如华为云码道CodeArts、Claude Code、Cursor等）是基于大语言模型的编程助手，能够：

| 能力      | 说明          | 示例                       |
| ------- | ----------- | ------------------------ |
| **读代码** | 理解现有代码结构和逻辑 | "分析memory-service.ts的功能" |
| **写代码** | 根据需求生成代码    | "实现里程碑检测功能"              |
| **改代码** | 修改、重构、修复代码  | "修复重复记录的bug"             |
| **查代码** | 搜索、定位代码     | "找到安全检查的实现位置"            |
| **跑命令** | 执行终端命令      | "npm run dev"            |
| **写文档** | 生成技术文档      | "记录今日工作进展"               |

### 2.2 Agent不能做什么

| 限制         | 说明            | 应对           |
| ---------- | ------------- | ------------ |
| **没有全局视野** | 每次会话上下文有限     | 主动提供项目背景     |
| **可能过度设计** | 倾向于"完整"而非"够用" | 明确约束"不要过度设计" |
| **可能理解偏差** | 对模糊需求可能理解错误   | 给出具体示例和约束    |
| **不会主动验证** | 写完代码不会自动测试    | 要求"写完就测试"    |
| **上下文会丢失** | 长会话可能遗忘早期信息   | 定期保存进展到文件    |

### 2.3 Agent的设计理念（以本项目为例）

本项目在与CodeArts Agent协同过程中，形成了以下设计理念：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Agent协同设计理念                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. 专业但不繁琐                                                     │
│     → Agent倾向完整，你要引导它"够用就好"                              │
│     → 本项目：五层架构 → 三层架构（实用主义）                          │
│                                                                     │
│  2. 渐进式建设                                                       │
│     → 不要一次让Agent实现所有功能                                     │
│     → 本项目：Phase 1→2→3→...，每Phase独立可用                       │
│                                                                     │
│  3. 人机兼顾                                                         │
│     → 让Agent生成的文件人也能看                                       │
│     → 本项目：profile.md（人可读）+ exercises.json（机器可读）         │
│                                                                     │
│  4. 先探讨再动手                                                     │
│     → 让Agent先分析方案，确认后再实现                                  │
│     → 本项目：先学习行业方案 → 对比分析 → 融合决策 → 分步实施          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 三、协同的核心原则

### 3.1 五大原则

#### 原则一：先给上下文，再给任务

**错误做法**：

```
你：实现记忆系统
Agent：[缺少项目背景，实现方向不确定]
```

**正确做法**：

```
你：回顾SPORTS-ASSISTANT项目，读取project-progress-2026-04-20.md，
    我们要开始长期记忆功能的开发
Agent：[理解项目背景，给出准确的分析和方案]
```

**本项目实例**（会话0）：

> 用户第一条消息就是"回顾SPORTS-ASSISTANT项目，回忆上次进度文件"，
> 让Agent先建立完整上下文，再开始工作。

---

#### 原则二：先探讨方案，再动手实现

**错误做法**：

```
你：按这个方案实现记忆系统
Agent：[直接写代码，可能方向错误]
```

**正确做法**：

```
你：学习这个方案文件，探讨改进与完善方案，
    不要过度设计，也不能不够专业
Agent：[分析方案，给出改进建议，等你确认]
你：融合两个方案，以v3.0为主，借鉴v2.0精华
Agent：[按你的决策方向实现]
```

**本项目实例**（会话0）：

> 用户让Agent学习`13-记忆系统完整方案.md`，先探讨改进方案。
> Agent分析后给出6个Phase的优先级排序。
> 用户再让Agent对比两个方案版本，Agent倾向v3.0实用主义版。
> 用户最终决策"以v3.0为主，借鉴v2.0精华"，Agent按此执行。

**关键洞察**：

> **你是决策者，Agent是分析者。**
> 让Agent提供分析和选项，你做最终决策。

---

#### 原则三：大任务拆小步骤，做完一步标一步

**错误做法**：

```
你：把记忆系统全部实现
Agent：[一次性实现，容易出错，难以验证]
```

**正确做法**：

```
你：制定改造步骤，写到文件里，按步骤执行，
    做完一步回文件标注，再往下进行
Agent：[Step 1→标注完成→Step 2→标注完成→...]
```

**本项目实例**（会话0）：

> 用户要求"制定步骤，写到文件，按步骤执行，做完标注"。
> Agent制定了11个Step，写入`记忆系统改造实施计划.md`。
> 每完成一个Step，更新文件标注状态。
> Step 2较大时，用户要求"分阶段完成，阶段划分写到文件"。

**关键洞察**：

> **可追踪的进度 = 写到文件 + 逐步标注。**
> 这比口头说"做完了"可靠得多。

---

#### 原则四：验证每个阶段的输出

**错误做法**：

```
你：实现里程碑检测
Agent：[写完代码就结束]
你：实现决策记录
Agent：[继续写，但上一步可能有问题]
```

**正确做法**：

```
你：实现里程碑检测，然后测试
Agent：[写代码→测试→发现问题→修复→再测试]
你：测试通过了，继续下一步
```

**本项目实例**（会话1）：

> 用户问"怎么测试这部分都做了，还有效果？"
> Agent创建测试文件，运行测试，全部通过。
> 但启动应用后发现"记录了没告诉用户"，立即修复。
> 又发现"里程碑消息格式不对"，再次修复。

**关键洞察**：

> **代码写完≠功能完成。必须启动应用，实际测试。**
> Agent写的代码可能有逻辑正确但体验不对的问题。

---

#### 原则五：保存上下文，方便恢复

**错误做法**：

```
[会话结束]
[第二天新会话]
你：继续昨天的项目
Agent：[没有上下文，不知道昨天做了什么]
```

**正确做法**：

```
[会话结束前]
你：整理今天的工作，记录成文件以便明天恢复上下文
Agent：[创建project-progress文件]

[第二天新会话]
你：回顾项目，读取project-progress文件
Agent：[快速恢复上下文，继续工作]
```

**本项目实例**（会话1→2）：

> 会话1结束前，用户要求"整理今天工作，记录成文件"。
> Agent创建`project-progress-2026-04-21.md`。
> 会话2开始时，用户直接给出恢复命令：
> `cat docs/project-progress-2026-04-21.md && npm run dev`

**关键洞察**：

> **好的进展文档 = 下次会话的上下文恢复点。**
> 包含：做了什么、文件在哪、怎么测试、下一步做什么。

---

### 3.2 原则速查表

| 原则   | 一句话         | 关键动作             |
| ---- | ----------- | ---------------- |
| 先上下文 | 先让Agent了解项目 | 读取项目文件、进展文档      |
| 先探讨  | 先分析再动手      | "先别动手写代码，探讨方案"   |
| 拆步骤  | 大任务拆小步骤     | "制定步骤，写到文件，做完标注" |
| 要验证  | 写完必须测试      | "启动应用，实际测试"      |
| 存上下文 | 结束前保存进展     | "整理工作，记录成文件"     |

## 四、实战案例：记忆系统改造

### 4.1 项目背景

**SPORTS-ASSISTANT**：面向50+女性的运动健康助手，核心特色"聊天即操作"。

**改造目标**：将记忆系统从简单的JSON存储，升级为具有行为阶段感知、信任等级演化、里程碑检测的智能记忆系统。

### 4.2 协同全过程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    人-Agent协同全过程                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  第1步：建立上下文                                                   │
│  人：回顾项目，读取进度文件                                           │
│  Agent：分析项目状态，确认当前阶段                                    │
│                                                                     │
│  第2步：学习行业方案                                                 │
│  人：学习这个方案文件，探讨改进，不要过度设计                          │
│  Agent：深度分析，给出6个Phase优先级排序                              │
│                                                                     │
│  第3步：对比决策                                                     │
│  人：比较两个方案版本，分析倾向                                       │
│  Agent：倾向v3.0实用主义版，给出5大理由                              │
│  人：融合，以v3.0为主，借鉴v2.0精华                                  │
│                                                                     │
│  第4步：制定步骤                                                     │
│  人：制定改造步骤，写到文件，按步骤执行                               │
│  Agent：制定11个Step，写入实施计划文件                               │
│                                                                     │
│  第5步：逐步实施                                                     │
│  人：按步骤执行，做完标注                                            │
│  Agent：Step1→Step2→...→Step11，每步更新文件                        │
│  人：Step2较大，分阶段写文件                                         │
│  Agent：Step2拆为4个子阶段，逐步执行                                 │
│                                                                     │
│  第6步：验证测试                                                     │
│  人：怎么测试？启动应用                                              │
│  Agent：创建测试文件，运行测试                                       │
│  人：记录了没告诉用户，里程碑格式不对                                 │
│  Agent：修复→再测试→修复→再测试                                     │
│                                                                     │
│  第7步：保存进展                                                     │
│  人：整理工作，记录成文件                                            │
│  Agent：创建进展文档，包含恢复命令                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.3 关键决策点

| 决策点  | Agent的建议    | 人的决策          | 理由            |
| ---- | ----------- | ------------- | ------------- |
| 架构选择 | 五层架构（学术完整）  | 三层架构（实用主义）    | YAGNI原则，先实现核心 |
| 文件格式 | 全JSON       | MD+JSON混合     | 人机兼顾，可读可维护    |
| 实施顺序 | 6个Phase并行   | Phase2→3先做    | 渐进式，每步独立可用    |
| 方案融合 | v2.0或v3.0选一 | 以v3.0为主借鉴v2.0 | 取长补短          |
| 过度设计 | 元认知、语义检索    | 后期再做          | 当前阶段价值不大      |

**核心洞察**：

> **每个关键决策点，都是人做最终决定。**
> Agent提供分析和选项，人根据项目实际情况决策。
> 这就是"协同"的本质——不是Agent替你决定，而是**Agent帮你做更好的决定**。

## 五、高效协同的沟通模式

### 5.1 给Agent的指令模板

#### 模式一：探索分析型

```
学习/分析 [文件/方案]，探讨 [改进方向]，
不要过度设计，也不能不够专业
```

**示例**：

> "学习13-记忆系统完整方案.md，探讨改进与完善方案，
> 不要过度设计，也不能不够专业。我们的工作要代表行业领先水平。"

**要点**：

- 明确输入（学习什么）
- 明确输出（探讨什么）
- 给出约束（不要过度设计，但要专业）

---

#### 模式二：对比决策型

```
比较 [方案A] 和 [方案B]，分析倾向哪种设计，给出理由
```

**示例**：

> "比较这两个方案，分析更倾向于哪种设计，并给出理由。"

**要点**：

- 让Agent做分析，你做决策
- 要求给出理由，不是简单选一个

---

#### 模式三：分步实施型

```
制定 [任务] 的步骤，写到文件里，
按步骤执行，做完回文件标注，再往下进行
```

**示例**：

> "制定改造步骤，写到文件里，然后按步骤去做，
> 做了就回去文件标注已做，再往下进行，直到全部完成。"

**要点**：

- 要求写到文件（可追踪）
- 要求逐步标注（可验证）
- 大步骤可再拆分

---

#### 模式四：验证修复型

```
启动应用，测试 [功能]，[问题描述]应该 [期望行为]
```

**示例**：

> "记录了没有告诉用户你记录了。之前不是纠正过这个问题吗？"
> "里程碑消息格式不对"

**要点**：

- 具体描述问题
- 说明期望行为
- Agent会定位问题并修复

---

#### 模式五：上下文保存型

```
整理今天的工作，记录成文件以便 [下次] 恢复项目上下文
```

**示例**：

> "整理一下今天的工作，记录成文件以便明天能快速恢复项目上下文"

**要点**：

- 目的明确：方便恢复
- Agent会包含：做了什么、文件在哪、怎么测试、下一步

---

### 5.2 沟通的反模式（不要这样做）

| 反模式     | 问题                 | 正确做法        |
| ------- | ------------------ | ----------- |
| "帮我做XX" | 太模糊，Agent理解可能偏差    | 具体描述需求+约束   |
| "全部实现"  | 一次性太多，容易出错         | 分步骤，逐步实施    |
| "就这样"   | 不验证，可能有问题          | 每步验证，启动测试   |
| "继续"    | 没有上下文，Agent不知道继续什么 | 先恢复上下文      |
| "改一下"   | 改什么？怎么改？           | 具体说明修改内容和期望 |

## 六、常见误区与避坑指南

### 6.1 五大误区

#### 误区一：让Agent一次性实现大功能

**问题**：Agent一次性实现大功能，代码量大，难以验证，出错概率高。

**案例**：如果让Agent一次性实现整个记忆系统（11个Step），很可能：

- 代码之间不一致
- 某些Step的实现方向错误
- 出错后难以定位问题

**正确做法**：分步实施，每步验证。

---

#### 误区二：不验证Agent的输出

**问题**：Agent写完代码就认为完成了，但可能有：

- 逻辑正确但体验不对（如"记录了没告诉用户"）
- 格式不对（如"里程碑消息格式错误"）
- 集成问题（如"重复记录"）

**案例**（本项目实际遇到）：

1. 运动记录后没有显示"已记录"提示
2. 里程碑消息格式不正确
3. 同一运动被重复记录

**正确做法**：启动应用，实际测试，发现即修复。

---

#### 误区三：不保存项目上下文

**问题**：会话结束后上下文丢失，下次会话Agent不知道之前做了什么。

**正确做法**：每次会话结束前，让Agent整理工作进展，写入文件。下次会话开始时，先读取进展文件恢复上下文。

---

#### 误区四：让Agent做所有决策

**问题**：Agent可能选择"最完整"的方案，而非"最合适"的方案。

**案例**：Agent分析后倾向五层架构（学术完整），但用户判断三层架构更合适（实用主义）。

**正确做法**：让Agent提供分析和选项，你做最终决策。特别是架构选择、优先级排序、范围控制等关键决策。

---

#### 误区五：不给约束条件

**问题**：没有约束，Agent可能过度设计或方向偏差。

**案例**：

- 不说"不要过度设计" → Agent实现20个数据结构
- 不说"先核心功能" → Agent同时实现所有功能
- 不说"代表行业领先水平" → Agent实现过于简陋

**正确做法**：给出明确的约束和期望。

---

### 6.2 避坑速查表

| 坑         | 信号                | 避坑方法        |
| --------- | ----------------- | ----------- |
| 一次做太多     | Agent输出大量代码       | 拆步骤，逐步实施    |
| 不验证       | "做完了"但没测试         | 启动应用，实际测试   |
| 上下文丢失     | 新会话Agent不知道之前做了什么 | 保存进展到文件     |
| Agent过度设计 | 代码结构过于复杂          | 明确"不要过度设计"  |
| 方向偏差      | 实现方向与预期不符         | 先探讨方案，确认再动手 |

## 七、项目工作流

### 7.1 标准工作流

```
┌─────────────────────────────────────────────────────────────────────┐
│                    人-Agent标准工作流                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐     │
│  │ 1.恢复    │───→│ 2.探讨    │───→│ 3.决策    │───→│ 4.规划    │     │
```

### 7.2 每步的关键动作

| 步骤      | 人的动作      | Agent的动作  | 产出      |
| ------- | --------- | --------- | ------- |
| 1.恢复上下文 | 给出进展文件路径  | 读取并分析     | 项目状态理解  |
| 2.探讨方案  | 提出需求+约束   | 分析方案，给出选项 | 方案分析报告  |
| 3.决策方向  | 做最终决策     | 提供理由和建议   | 确定的方向   |
| 4.规划步骤  | 要求写到文件    | 制定步骤，写入文件 | 实施计划文件  |
| 5.实施验证  | 要求逐步执行+测试 | 执行+测试+标注  | 代码+测试结果 |
| 6.修复问题  | 描述具体问题    | 定位+修复     | 修复后的代码  |
| 7.整理文档  | 要求整理工作    | 生成进展文档    | 进展文档    |
| 8.保存进展  | 确认保存      | 写入文件      | 可恢复的上下文 |

### 7.3 本项目的实际工作流

| 会话  | 主要工作                 | 关键协同模式             |
| --- | -------------------- | ------------------ |
| 会话0 | 方案学习→对比→决策→制定步骤→开始实施 | 探索分析型+对比决策型+分步实施型  |
| 会话1 | 继续实施→测试→修复问题→保存进展    | 分步实施型+验证修复型+上下文保存型 |
| 会话2 | 恢复上下文→继续开发→整理文档      | 上下文恢复+分步实施型        |

## 八、如何保持项目记忆

> 原则五"保存上下文"只讲了"写进展文件"，但实际项目中，保持记忆远不止于此。

### 8.1 项目记忆的层次

```
┌─────────────────────────────────────────────────────────────────────┐
│                    项目记忆的三个层次                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  第一层：会话记忆（短期）                                              │
│  → 当前会话中的对话上下文                                              │
│  → 会话结束即丢失                                                     │
│  → 用途：当前任务的连续对话                                            │
│                                                                     │
│  第二层：进展记忆（中期）                                              │
│  → 写入文件的进展文档、实施计划                                        │
│  → 跨会话可恢复                                                      │
│  → 用途：跨会话的连续开发                                              │
│                                                                     │
│  第三层：决策记忆（长期）                                              │
│  → 架构决策、技术选型、设计约束                                        │
│  → 项目生命周期内持续有效                                              │
│  → 用途：保持项目一致性，防止Agent偏离方向                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 8.2 进展记忆的结构化模板

一份好的进展文档应包含以下要素：

```markdown
# 项目进展 YYYY-MM-DD

## 今日完成
- 具体做了什么（含文件路径和行号）

## 待验证问题
- 修改后是否生效？如何验证？

## 关键决策
- 为什么选择A而不是B？

## 关键文件变更
| 文件 | 变更内容 |
|------|----------|

## 下一步
- 明天继续做什么？

## 恢复命令
- 启动服务器、测试方法
```

**本项目实例**：

> `project-progress-2026-04-20.md` 记录了意图规则优化、运动记录查询、数据修正，
> 包含待验证问题（数据修改未生效），下次会话可直接定位问题。

### 8.3 决策记忆：防止Agent偏离方向

**问题**：Agent没有长期记忆，下次会话可能做出与之前决策矛盾的选择。

**案例**：

- 会话1决定"三层架构（实用主义）"
- 会话2如果不说，Agent可能倾向"五层架构（学术完整）"

**解决方法**：

| 方法        | 说明             | 示例                    |
| --------- | -------------- | --------------------- |
| **决策日志**  | 在项目中维护一份决策记录文件 | `docs/decisions.md`   |
| **恢复时强调** | 恢复上下文时重申关键决策   | "我们选择了三层架构"           |
| **约束文件**  | 将约束写入项目配置      | `.arts/settings.json` |

**决策日志模板**：

```markdown
# 项目决策记录

## DEC-001: 架构选择
- **决策**：三层架构（实用主义），非五层
- **理由**：YAGNI原则，先实现核心
- **日期**：2026-04-20
- **状态**：已确认

## DEC-002: 文件格式
- **决策**：MD+JSON混合
- **理由**：人机兼顾，可读可维护
- **日期**：2026-04-20
- **状态**：已确认
```

### 8.4 记忆的版本管理

```
project-progress-2026-04-20.md  ← 每日一份
project-progress-2026-04-21.md
project-progress-2026-04-22.md
```

**组织原则**：

- 每次会话结束前创建/更新当天的进展文件
- 文件名含日期，方便按时间查找
- 新会话恢复时，读取**最近一份**进展文件即可
- 关键决策单独维护在 `docs/decisions.md`，不随日期归档

### 8.5 运行时状态与文件记忆的区别

**本项目踩过的坑**：

> 修改了JSON数据文件（广场舞→舞蹈，删除力量训练），
> 但查询仍返回旧数据。原因是服务器缓存了旧数据，需要重启。

**教训**：

- **文件修改 ≠ 运行时生效**：修改数据文件后，需要重启服务或刷新缓存
- **记忆有两层**：文件层（持久化）+ 运行时层（内存缓存）
- **验证时要注意**：修改文件后，确保运行时也重新加载

## 十、设计方案准备方法

> 原则二"先探讨再动手"讲了怎么让Agent分析方案，但没讲怎么准备方案。
> 方案准备是人的核心工作——好的方案文档让Agent高效工作，差的方案让Agent反复返工。

### 10.1 方案文档的作用

```
┌─────────────────────────────────────────────────────────────────────┐
│                    方案文档在协同中的位置                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  人 ──→ 方案文档 ──→ Agent ──→ 代码                                  │
│                                                                     │
│  方案文档是人与Agent之间的"接口契约"：                                  │
│  - 人：通过方案文档表达意图和约束                                       │
│  - Agent：通过方案文档理解需求和边界                                    │
│  - 代码：是方案文档的可执行实现                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 10.2 方案文档的结构模板

一份好的方案文档应包含：

```markdown
# [功能名称] 设计方案

## 1. 需求描述
- 要解决什么问题？
- 用户场景是什么？

## 2. 核心设计
- 架构图/流程图
- 关键数据结构
- 核心接口定义

## 3. 约束条件
- 不要做什么（防止过度设计）
- 必须满足什么（防止不够专业）
- 技术限制（语言、框架、兼容性）

## 4. 实施步骤
- Phase 1: 最小可用版本
- Phase 2: 增强功能
- Phase 3: 优化完善

## 5. 验证标准
- 怎么判断做对了？
- 测试用例/验收标准
```

### 10.3 本项目的方案准备实例

**实际做法**：用户准备了两个方案版本

| 方案          | 特点        | Agent的评价 |
| ----------- | --------- | -------- |
| v2.0（学术完整版） | 五层架构，功能全面 | 完整但过度设计  |
| v3.0（实用主义版） | 三层架构，核心优先 | 实用且可渐进   |

**关键操作**：

1. 人准备两个方案 → 让Agent对比分析
2. Agent倾向v3.0 → 人做最终决策"以v3.0为主，借鉴v2.0精华"
3. 融合方案 → Agent按融合方案实施

**洞察**：

> **准备多个方案比只准备一个好。**
> 单一方案Agent只能"是/否"，多个方案Agent能对比分析，你做融合决策。

### 10.4 需求文档怎么写（Agent才能理解）

| 要素      | 好的写法                        | 差的写法       |
| ------- | --------------------------- | ---------- |
| **具体性** | "用户说'今天做了什么运动'时，返回今天的运动记录"  | "查询运动记录"   |
| **约束性** | "不要过度设计，三层架构即可"             | "设计一个记忆系统" |
| **示例性** | "输入：'明天做什么运动' → 输出：明天的运动计划" | "支持运动查询"   |
| **边界性** | "只支持今天/昨天/本周三种时间范围"         | "支持时间查询"   |
| **优先级** | "Phase1先做记录存储，Phase2再做智能推荐" | "全部实现"     |

### 10.5 技术选型怎么表达

让Agent做技术对比时，给出评估维度：

```
比较 [方案A] 和 [方案B]，从以下维度分析：
- 实现复杂度
- 维护成本
- 扩展性
- 与现有架构的兼容性
- 当前阶段的适用性
```

**本项目实例**：

> 用户让Agent对比v2.0和v3.0，Agent从5个维度分析后倾向v3.0。
> 评估维度越明确，Agent的分析越有针对性。

## 十一、与其它Agent协作

> 当前课程只讲了"人与一个Agent"的协同，实际项目中可能涉及多个Agent。

### 11.1 多Agent协作的场景

```
┌─────────────────────────────────────────────────────────────────────┐
│                    多Agent协作的典型场景                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  场景一：专业分工                                                    │
│  → Agent-A：写代码（CodeArts/Claude Code）                           │
│  → Agent-B：写测试（专用测试Agent）                                   │
│  → Agent-C：写文档（专用文档Agent）                                   │
│                                                                     │
│  场景二：流水线                                                      │
│  → Agent-A：需求分析 → 生成设计文档                                   │
│  → Agent-B：根据设计文档 → 生成代码                                   │
│  → Agent-C：代码审查 → 反馈修改建议                                   │
│                                                                     │
│  场景三：对比验证                                                    │
│  → Agent-A：实现方案A                                                │
│  → Agent-B：实现方案B                                                │
│  → 人：对比两个实现，选择或融合                                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 11.2 Agent间上下文传递

**核心问题**：Agent-A的输出如何成为Agent-B的输入？

| 传递方式     | 说明                       | 适用场景          |
| -------- | ------------------------ | ------------- |
| **文件传递** | Agent-A写入文件，Agent-B读取文件  | 设计文档→代码实现     |
| **指令传递** | 人在指令中引用Agent-A的输出        | "根据刚才的设计文档实现" |
| **代码传递** | Agent-A写代码，Agent-B基于代码工作 | 代码实现→代码审查     |

**最佳实践**：

> **文件是最可靠的传递媒介。**
> Agent-A的输出写入文件 → Agent-B读取文件 → 上下文不会丢失。

### 11.3 分工与边界

| 原则        | 说明               | 反例                     |
| --------- | ---------------- | ---------------------- |
| **单一职责**  | 每个Agent负责一个领域    | 让写代码的Agent也写文档         |
| **明确接口**  | Agent间通过文件/接口交互  | Agent间通过口头描述传递         |
| **人做协调**  | 人负责分配任务和整合结果     | 让Agent自己决定分工           |
| **结果可验证** | 每个Agent的输出都可独立验证 | Agent-A的输出只有Agent-B能理解 |

### 11.4 当前项目的协作模式

本项目目前是**人与单Agent**协作，但可以扩展：

```
当前模式：
  人 ←→ CodeArts Agent（写代码+测试+文档）

可扩展模式：
  人 ←→ CodeArts Agent（写代码）
       ←→ 测试Agent（自动化测试）
       ←→ 文档Agent（生成文档）
```

**扩展建议**：

- 先做好人与单Agent的协同（当前课程已覆盖）
- 再逐步引入其他Agent，每次只加一个
- 每个新Agent都要建立上下文恢复机制

## 十二、指令风格总结与优缺点分析

> 基于本项目实际协同过程，总结用户的指令风格特征。

### 12.1 风格特征

| 特征         | 表现               | 示例                 |
| ---------- | ---------------- | ------------------ |
| **目标驱动**   | 先说目标，再说方法        | "记录今天的内容，明天继续"     |
| **边测边改**   | 启动后立即测试，发现问题立即反馈 | "你看以下回复就不对"        |
| **直接指出问题** | 不绕弯，直接说哪里不对      | "45秒应该45分钟，单位不对"   |
| **要求智能**   | 期望Agent自己判断      | "你要智能一点哇"          |
| **约束双方向**  | 既防过度又防不足         | "不要过度设计，也不能不够专业"   |
| **要求可追踪**  | 要求写到文件、标注进度      | "做完回文件标注"          |
| **人做决策**   | 关键决策自己下          | "以v3.0为主，借鉴v2.0精华" |

### 12.2 优点

1. **反馈极快**：测试后立即指出问题，Agent能快速定位修复。这是最高效的协同方式——比写长篇需求文档快得多

2. **约束精准**："不要过度设计，也不能不够专业"这种双向约束，比单方向约束效果好很多，Agent不会走极端

3. **决策清晰**：关键决策点直接下判断（"融合两个方案"），不拖泥带水，Agent执行方向明确

4. **要求可追踪**：要求"写到文件、做完标注"，这让进度可视化，不会出现"口头说做完但实际没做完"的情况

5. **先探讨后动手**：不让Agent上来就写代码，先分析方案，减少返工

### 12.3 可改进之处

1. **有时期望Agent"自己判断"**：
   
   - "你要智能一点哇"——Agent确实需要更智能，但更好的做法是**给出判断规则**
   - 改进：不说"智能一点"，而说"做了=查记录，做=查计划"
   - 本项目后来实际上也是这么做的（添加正则规则），但可以更早明确

2. **问题描述可以更结构化**：
   
   - 有时只说"不对"，可以加上"期望是什么"
   - 好的示例："45秒应该45分钟"（有问题+有期望）
   - 差的示例："你看以下回复就不对"（缺少期望描述）

3. **缺少预防性指令**：
   
   - 大多是发现问题后修复，可以提前给约束
   - 示例："修改数据后要重启服务器确保生效"
   - 本项目数据修改未生效就是这个问题

4. **上下文恢复可以更自动化**：
   
   - 目前手动说"回顾项目，读取进度文件"
   - 可以准备一个标准恢复脚本/命令

### 12.4 风格评分

| 维度    | 评分(1-5) | 说明               |
| ----- | ------- | ---------------- |
| 反馈速度  | 5       | 测试后秒级反馈，最高效的协同方式 |
| 约束质量  | 4       | 双向约束很好，偶尔缺预防性约束  |
| 决策效率  | 5       | 关键决策果断清晰，不拖泥带水   |
| 问题描述  | 3.5     | 有时只说"不对"，缺少期望描述  |
| 进度管理  | 4       | 要求可追踪，但可以更早建立    |
| 上下文管理 | 4       | 知道保存和恢复，但可以更结构化  |

## 补充实战案例：历史会话导出导入功能——从方案演进到迭代实现

> 本案例完整记录了"历史会话"功能从Day1的方案尝试、到Day2的方案推翻重来、再到迭代打磨的全过程。
> 这是一个典型的**人-Agent协同中"方案演进"**的案例——Agent提出了初始方案，人在使用中发现问题，与Agent协商后推翻原方案、建立新方案，并通过分步迭代实现最终目标。

### 背景

SPORTS-ASSISTANT项目的前端对话历史存储在浏览器localStorage中，页面刷新后保留，但换设备/换浏览器后历史丢失。需要实现**跨设备的对话历史迁移**能力。

---

### Day 1（2026-04-26）：服务端持久化方案

#### 人-Agent对话过程

```
人：对话历史只存在内存中，页面刷新/服务器重启后丢失，需要持久化
Agent：提出方案——在服务端持久化对话历史
       1. MemoryService 添加 saveChatMessage() 和 getChatHistory()
       2. 对话按日保存到 data/users/{userId}/chats/{date}.json
       3. 添加 connectionUserMap 映射
       4. 提供 GET /api/chat-history 和 POST /api/chat-history/sync 接口
       5. 前端页面加载时自动同步 localStorage → 服务器
人：可以，实现吧
Agent：实现并部署到公网
人：测试一下……本地浏览器的历史在公网浏览器看不到
Agent：这是因为localStorage是浏览器本地存储，不同设备的localStorage互不相通
       需要增加导出/导入功能
人：好的，明天继续
```

#### Day 1 实现的代码

| 组件    | 代码                                | 说明                        |
| ----- | --------------------------------- | ------------------------- |
| 后端持久化 | `MemoryService.saveChatMessage()` | 按日保存到 `chats/{date}.json` |
| 后端读取  | `MemoryService.getChatHistory()`  | 读取多日历史，返回最近N条             |
| API路由 | `GET /api/chat-history`           | 获取用户对话历史                  |
| API路由 | `POST /api/chat-history/sync`     | 同步本地历史到服务器                |
| 前端同步  | `loadServerChatHistory()`         | 从服务器加载历史                  |
| 前端同步  | `syncLocalHistoryToServer()`      | 把localStorage历史POST到服务器   |

#### Day 1 的问题

**核心矛盾**：服务端持久化方案假设"用户的对话历史会自动流向服务器"，但实际上：

- 不同设备的localStorage互不相通——本地电脑的历史无法出现在公网浏览器
- 页面加载时自动同步，只能同步**当前浏览器**已有的历史
- 用户没有"主动迁移"历史的手段

**结论**：服务端自动同步方案**只能解决"服务器重启后恢复"的问题**，无法解决"跨设备迁移"的核心需求。

---

### Day 2（2026-04-27）：推翻原方案，建立导出/导入方案

#### 早晨：回顾与规划

```
人：回顾项目，继续推进，你上次写的日志在 xxx
Agent：[读取日志] 总结昨天完成和未完成的工作
       高优先级：实现对话历史导出/导入功能
人：实现历史对话的导出导入功能，就在每个会话右边三个点里增加菜单，
    你先做个步骤计划，一步步实现，本地实现导出后让我测试，
    测试通过再做导入，测试通过我们再同步公网，并清理掉昨天做的无用代码
```

#### 关键指令分析

这条指令包含了多个协同要素：

| 要素        | 内容               | 作用                   |
| --------- | ---------------- | -------------------- |
| **功能位置**  | "每个会话右边三个点里增加菜单" | 精确指定UI位置，避免Agent自由发挥 |
| **分步计划**  | "先做个步骤计划"        | 要求Agent先规划再执行        |
| **迭代验证**  | "导出→测试→导入→测试→公网" | 每步验证，不跳步             |
| **清理旧代码** | "清理掉昨天做的无用代码"    | 保持代码整洁，删除废弃方案        |

#### Step 1-2：实现导出功能

```
Agent：制定8步计划，开始实现导出
       - 在三点菜单中添加"导出"菜单项
       - 实现 exportChat() 函数，将消息导出为JSON文件下载
人：[测试] 正确导出了对话记录 chat-___长期记忆-2026-04-27.json ✓
```

#### Step 3-4：实现导入功能——**踩坑与修复**

```
Agent：实现 importChat() 函数
人：[测试] 导入成功！新增 0 条消息（按timestamp去重）
     新建对话导入的，是没导成功，不是去重
```

**Bug 1：消息格式不一致**

- 存储格式：`{ text, type, timestamp }`（前端 `updateCurrentChat` 使用的格式）
- 导入检查：`msg.role && msg.content`（按服务端API格式检查）
- 结果：所有消息的 `role` 和 `content` 都是 `undefined`，导入全部失败

**修复**：导出和导入都适配两种格式：

- 导出时：`text = m.text || m.content`，`type = m.type || m.role`
- 导入时：同样做双格式适配

```
人：[测试] 导入成功了，但只有用户的问题，没有回复
```

**Bug 2：去重键设计不当**

- `updateCurrentChat()` 给 user 和 assistant 消息分配了**相同的 timestamp**（`Date.now()`）
- 去重用 `timestamp` 作为唯一键 → 同一时刻的 assistant 回复被误判为重复消息
- 结果：导入后只剩用户消息，助手回复全丢了

**修复**：去重键从 `timestamp` 改为 `timestamp_type`，同一时刻不同角色的消息不会被误去重

```
人：[测试] 导入也可以了 ✓
```

#### Step 5：增加Markdown导出

```
人：导出增加Markdown格式，方便用户查看。导入可以不增加。
Agent：在三点菜单中增加"导出JSON"和"导出Markdown"两个选项
       Markdown格式包含标题、时间、用户/助手对话、消息计数
人：[测试] MD可以了 ✓
```

#### Step 6-7：公网部署与旧代码清理

```
Agent：上传 index.html 到服务器，pm2 restart，验证 health ✓
人：清理昨天无用的代码
Agent：[确认清理范围] 全删（推荐）
       删除前端 loadServerChatHistory / syncLocalHistoryToServer
       删除后端 /api/chat-history 和 /api/chat-history/sync 路由
       删除 connectionUserMap
       删除 addToHistory 中的 saveChatMessage 调用
       删除 MemoryService 中 saveChatMessage / getChatHistory 方法
       编译通过，部署到公网 ✓
```

### 协同经验总结

#### 1. 方案演进是常态，不必追求一次到位

```
Day 1 方案：服务端自动同步（saveChatMessage + API + 前端自动同步）
    ↓ 使用中发现问题
Day 2 方案：客户端手动导出/导入（exportChat + importChat）
    ↓ 迭代打磨
最终方案：导出JSON + 导出Markdown + 导入JSON（三点菜单）
```

> **经验**：Agent提出的第一个方案不一定是最终方案。人与Agent协同的价值在于——
> Agent快速实现方案 → 人在使用中发现问题 → 与Agent协商调整 → 迭代到正确方案。
> 这个循环比"人在脑子里想清楚再让Agent实现"更高效。

#### 2. "先测试再继续"的迭代节奏

本次实现严格遵循了用户要求的迭代节奏：

```
实现导出 → 测试导出 → 实现导入 → 测试导入(发现问题) → 修复 → 再测试 → 增加MD → 测试 → 部署 → 清理
```

每一步都有人参与验证，避免了"一次性写完所有代码 → 发现一堆bug → 大返工"的风险。

> **经验**：分步迭代不是慢，而是稳。特别是涉及数据格式、去重逻辑等容易出错的场景，
> 边写边测比写完再测效率更高。

#### 3. 数据格式Bug的教训

两次Bug都与**数据格式**有关：

| Bug   | 原因                                      | 教训                           |
| ----- | --------------------------------------- | ---------------------------- |
| 格式不一致 | 前端用 `{text,type}`，导入检查 `{role,content}` | **Agent写新功能时，必须先读现有代码的存储格式** |
| 去重键不当 | 同一时刻的 user+assistant 共享 timestamp       | **去重键设计要考虑数据生成的实际行为**        |

> **经验**：Agent容易"假设"数据格式，而不是"确认"数据格式。
> 人在审查代码时，应特别关注**新旧代码的接口约定是否一致**。

#### 4. 清理旧代码的时机

Day 1 的服务端同步代码在 Day 2 被完全删除。什么时候清理？

> **经验**：当新方案已验证可用时，立即清理旧方案代码。
> 不要"先留着备用"——旧代码会增加维护负担和认知负荷。
> 如果真的需要回退，git历史里有。

#### 5. 精确指令的价值

对比两条指令的效果：

| 指令                                         | 效果                                 |
| ------------------------------------------ | ---------------------------------- |
| "实现导出导入功能"                                 | Agent可能做成后端API、可能做成文件按钮、可能一步写完所有代码 |
| "在每个会话右边三个点里增加菜单，先做步骤计划，导出→测试→导入→测试→公网→清理" | Agent精确执行，分步迭代，每步验证                |

> **经验**：给Agent的指令越精确，输出越可控。特别是：
> 
> - **位置**："三点菜单里"比"添加按钮"好
> - **顺序**："导出→测试→导入→测试"比"把导出导入都实现"好
> - **约束**："清理旧代码"比"完成功能"更完整

#### 6. 人的角色：决策者与测试者

在整个过程中，人的核心角色是：

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  Day 1：决策者 —— 批准Agent的服务端同步方案          │
│         测试者 —— 发现跨设备历史不通的问题           │
│                                                     │
│  Day 2：规划者 —— 指定功能位置、迭代节奏、清理要求   │
│         测试者 —— 每步验证，发现两个Bug              │
│         决策者 —— 选择"全删"而非"保留备用"           │
│                                                     │
└─────────────────────────────────────────────────────┘
```

> **经验**：人是质量的最终把关者。Agent能快速产出代码，但代码是否正确、
> 方案是否合理、格式是否一致——这些都需要人来验证和决策。
