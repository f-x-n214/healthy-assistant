# 意图识别模块 — 教学设计

## 课程定位

**前置课程**：智能对话（第1-2周）、超越对话框思想（第6周）  
**本节目标**：在智能对话基础上，加入"理解用户想做什么"的能力  
**后续课程**：记忆系统 → 工具执行

---

## 一、核心问题

### 1.1 现状

现在，你的助手能对话了：

```
用户：我今天散步了30分钟
助手：很好！运动对身体有益，建议坚持下去。
```

看起来还行。但问题来了——助手只是在"说话"，它真的**理解**用户想做什么吗？

### 1.2 问题场景

```
场景1：助手不知道用户"记录运动"
用户：我今天散步了30分钟        → 助手可能只是回应"不错"
       第二天，用户问：我上周运动了多少？
       助手：不知道，因为上次说的话没有存下来 ❌

场景2：助手不知道用户"要推荐"
用户：最近想运动，但不知道做什么好
       助手：运动很好，要坚持。   → 没有给出具体建议 ❌

场景3：助手不知道用户"在问饮食"
用户：苹果有多少热量？
       助手：苹果是一种水果，富含维生素...  → 答非所问 ❌
```

### 1.3 本质问题

> **用户说的每一句话，都有一个"目的"。助手如果不知道这个目的，就只能泛泛而谈。**

意图识别，就是让助手**听懂用户想做什么**。

## 二、为什么需要意图识别

### 2.1 对比：没有意图识别 vs 有意图识别

| 对话 | 没有意图识别 | 有意图识别 |
|------|-------------|-----------|
| "我今天散步了30分钟" | "很好，继续加油" | **识别为：运动记录 → 自动存到数据库** |
| "苹果有多少热量" | "苹果富含维生素C" | **识别为：营养查询 → 返回热量数据** |
| "我有高血压" | "高血压要注意饮食" | **识别为：健康档案更新 → 自动更新用户档案** |

### 2.2 意图识别能做什么

```
意图识别 = 听懂了用户的目的
   │
   ├──→ 识别之后，才能做对的事
   │        用户要记录 → 存数据
   │        用户要查询 → 读数据
   │        用户要推荐 → 调用推荐逻辑
   │
   └──→ 识别之后，才能记住对的信息
            用户聊了运动 → 记住是"运动"相关
            用户聊了饮食 → 记住是"饮食"相关
```

### 2.3 教学案例：为什么不能直接问用户？

有的同学说：那我让用户自己选"你要记录运动还是查询营养"不就行了？

**不好。原因：**

| 问题 | 说明 |
|------|------|
| 增加用户负担 | 用户要学"怎么用" |
| 不够自然 | 生活中没人这么说话 |
| 失去主动性 | 助手变成"被动的" |

```
❌ 不自然的方式：
助手：请选择：1.记录运动 2.查询营养 3.健康档案
用户：1
助手：请输入运动类型
用户：散步
助手：请输入时长
用户：30分钟

✅ 自然的对话：
用户：我今天散步了30分钟
助手：（识别为"运动记录"）好的，已记录您今天散步30分钟！
```

## 三、意图识别设计方案

### 3.1 整体思路

**核心问题**：用户说一句话，怎么知道他的目的是什么？

**解决方案**：分层处理，从简单到复杂

```
用户输入
   │
   ├── 第一层：简单问题 → 直接回答（不调用AI）
   │
   └── 复杂问题 → 调用AI识别意图 → 执行对应能力
```

### 3.2 为什么分层？

**思考：如果所有问题都直接调用AI大模型，会怎么样？**

| 问题 | 调用AI的成本 |
|------|------------|
| "你好" | 浪费，每次都要调用 |
| "再见" | 浪费，每次都要调用 |
| "我今天散步了" | 值得调用 |
| "苹果热量多少" | 值得调用 |

**结论**：
- 简单、寒暄类的问题，**不值得**调用大模型
- 复杂、有实际需求的问题，**才值得**调用大模型

### 3.3 方案一：两层架构（最简单）

```
用户消息
   │
   ▼
正则匹配检查
   │
   ├── 匹配"你好"、"谢谢" → 直接返回问候语
   │
   └── 不匹配 → 调用大模型识别意图 → 执行
```

**代码逻辑**：

```javascript
// 第一层：正则匹配（快、省钱）
const fastPatterns = [
  { pattern: /^你好|^您好/, response: "您好！有什么可以帮您？" },
  { pattern: /^谢谢|^感谢/, response: "不客气！" },
  { pattern: /^再见|^拜拜/, response: "再见，记得每天运动哦！" },
];

for (const rule of fastPatterns) {
  if (rule.pattern.test(message)) {
    return rule.response;  // 不调用大模型，直接返回
  }
}

// 第二层：复杂问题才调用大模型
const intent = await callLLM(message);
return executeByIntent(intent);
```

**优点**：简单，响应快，成本低  
**缺点**：正则写死了，不够灵活

### 3.4 方案二：关键词路由（进阶）

**思路**：不写死正则，而是定义"意图关键词"

```
意图类型          关键词
─────────────────────────
运动记录    →   散步、跑步、游泳、瑜伽、打太极、广场舞...
运动查询    →   做什么运动好、推荐、适合、能不能...
饮食记录    →   吃了、早饭、午饭、晚饭...
营养查询    →   热量、脂肪、蛋白质、维生素...
健康更新    →   血压、血糖、高血压、糖尿病...
```

**代码逻辑**：

```javascript
// 关键词路由
const intentKeywords = {
  exercise_log: ['散步', '跑步', '游泳', '瑜伽', '太极'],
  diet_log: ['吃了', '早饭', '午饭', '晚饭', '午餐'],
  health_update: ['血压', '血糖', '高血压', '糖尿病'],
};

function classifyByKeywords(message) {
  for (const [intent, keywords] of Object.entries(intentKeywords)) {
    if (keywords.some(k => message.includes(k))) {
      return intent;
    }
  }
  return 'chat';  // 默认对话
}
```

**问题**：关键词有重叠怎么办？

```
用户说："我今天散步了，苹果有多少热量？"
   → 包含"散步" → 运动记录
   → 包含"热量" → 营养查询
   → 两个意图都有 ❌
```

### 3.5 方案三：AI识别 + 分类（我们的选择）

**思路**：对于复杂消息，调用大模型让它判断意图

```
用户消息：我今天散步了30分钟
   │
   ▼
构建Prompt：
"用户说了什么？判断他的意图是：
 A. 运动记录（想记录今天的运动）
 B. 运动查询（想问运动相关问题）
 C. 饮食记录（想记录吃了什么）
 D. 健康更新（想更新健康信息）
 E. 普通对话"
   │
   ▼
大模型返回：A（运动记录）
   │
   ▼
执行：提取"散步"、"30分钟" → 存入运动记录
```

**优点**：灵活、可以理解复杂语义  
**缺点**：需要调用大模型，有成本和延迟

### 3.6 最终方案：分层混合架构

综合以上，我们采用**分层处理**：

```
┌─────────────────────────────────────────────────────┐
│                    用户消息输入                       │
└─────────────────────┬───────────────────────────────┘
                      │
                      ▼
        ┌─────────────────────────┐
        │  第一层：快速匹配（零成本）│  <5毫秒
        │  正则 + 关键词           │
        └───────────┬─────────────┘
                    │
         ┌──────────┴──────────┐
         │ 匹配到？              │
         ▼                      ▼
        是                     否
        │                      │
        ▼                      ▼
   直接返回               ┌─────────────────────────┐
   问候/确认/告别         │  第二层：AI意图识别     │  500毫秒-1秒
                         │  调用大模型判断意图      │
                         └───────────┬─────────────┘
                                     │
                                     ▼
                          ┌─────────────────────┐
                          │  识别结果            │
                          │  - 意图类型          │
                          │  - 置信度            │
                          │  - 提取的关键信息    │
                          └───────────┬───────────┘
                                     │
                                     ▼
                          ┌─────────────────────┐
                          │  第三层：安全检查    │
                          │  这个意图安全吗？    │
                          └───────────┬───────────┘
                                     │
                          ┌──────────┴──────────┐
                          │  安全？              │
                          ▼                      ▼
                        是                     否
                         │                      │
                         ▼                      ▼
                   执行对应能力            拦截 + 提示
```

### 3.7 为什么这样设计？—— 设计决策解释

| 决策 | 为什么 | 类比 |
|------|--------|------|
| **第一层用正则，不用AI** | "你好"这种不需要理解，AI判断反而浪费 | 问路不需要查地图 |
| **第二层才用AI** | 复杂语义AI才能准确判断 | 复杂问题才需要专家 |
| **第三层做安全检查** | 健康领域容不得错误 | 医疗设备有安全阀 |
| **分层不是一层打天下** | 不同问题用不同方法，省钱又高效 | 走路不开车，开车不走路 |

## 四、意图类型设计

### 4.1 我们需要识别哪些意图？

根据50+女性运动健康场景，定义5个**能力域**：

| 能力域 | 用户想做什么 | 示例 |
|--------|------------|------|
| **运动陪伴** | 记录运动、查询运动、推荐运动 | "我今天散步了"、"做什么运动好" |
| **饮食助手** | 记录饮食、查询营养、饮食建议 | "今天吃了苹果"、"高血压怎么吃" |
| **健康档案** | 更新健康信息、查询健康状况 | "我有高血压"、"最近血压怎么样" |
| **数据看板** | 查看统计、趋势报告 | "我这周运动了多少" |
| **智能对话** | 聊天、问候、情绪关怀 | "你好"、"今天心情不好" |

### 4.2 意图层级结构

```
能力域
   │
   ├── 运动陪伴
   │      ├── 运动记录：我今天散步了30分钟
   │      ├── 运动查询：太极拳有什么好处
   │      └── 运动推荐：推荐适合我的运动
   │
   ├── 饮食助手
   │      ├── 饮食记录：今天吃了苹果
   │      ├── 营养查询：苹果有多少热量
   │      └── 饮食建议：高血压应该怎么吃
   │
   ├── 健康档案
   │      ├── 健康更新：我有高血压
   │      ├── 运动姿势拍照：帮我看看姿势对不对
   │      └── 取消运动：今天不想运动了
   │
   ├── 数据看板
   │      ├── 今日统计：看看今天的运动饮食
   │      ├── 周统计：这周运动怎么样？
   │      ├── 月统计：这个月坚持得怎么样？
   │      ├── 趋势分析：看看我三个月的变化
   │      └── 目标进度：我离目标还差多少？
   │
   └── 智能对话
          ├── 问候/告别：你好、再见
          ├── 感谢：谢谢
          ├── 情绪关怀：最近心情不好
          └── 鼓励表扬：（用户坚持得好，主动表扬）
```

### 4.3 完整意图分类表

| 大类 | 代码 | 子意图数 | 核心能力 |
|------|------|---------|---------|
| 运动陪伴 | exercise | 6 | 记录、查询、推荐、提醒、拍照、取消 |
| 饮食助手 | diet | 6 | 记录、查询、建议、拍照、提醒、对比 |
| 健康档案 | health | 5 | 更新、查询、症状、用药、咨询 |
| 数据看板 | stats | 5 | 日/周/月统计、趋势、目标进度 |
| 智能对话 | chat | 5 | 问候、闲聊、关怀、鼓励、帮助 |
| 提醒触发 | remind | 5 | 运动/饮食/用药/体检/目标提醒 |

**共6大类、30+种子意图**

### 4.4 设计决策：为什么这样分？

**问题**：为什么不分成更多更细的意图？比如"记录散步"、"记录跑步"、"记录游泳"？

**回答**：

| 分法 | 意图数 | 问题 |
|------|--------|------|
| 按具体动作分 | 20+个 | 太细，大模型也容易分错 |
| 按能力域分 | 6大类 | 清晰，容易判断 |

**原则**：
- 意图分类要**足够粗**，大模型容易判断
- 具体信息由**参数提取**完成，不占意图分类
- 子意图用下划线命名，如 `exercise_log`、`diet_photo`

### 4.5 存储 vs 提示词 —— 数据流向全流程

**核心原则**：存储什么格式，就提供什么格式给模型。格式不是给模型看的，是给代码看的。

```
┌──────────────────────────────────────────────────────────────┐
│  存储层（JSON / 数据库）                                      │
│  应用把数据存成结构化格式，好查、好改、好统计                    │
└─────────────────────────┬────────────────────────────────────┘
                          │  检索时取出
                          ▼
┌──────────────────────────────────────────────────────────────┐
│  提示词层（自然语言）                                         │
│  把检索结果拼成文字发给模型，模型只关心文字内容                    │
└─────────────────────────┬────────────────────────────────────┘
                          │  模型理解意图，返回结构化输出
                          ▼
┌──────────────────────────────────────────────────────────────┐
│  输出层（JSON）                                              │
│  模型的输出必须是JSON，因为代码要解析、处理、执行                 │
└──────────────────────────────────────────────────────────────┘
```

| 层级 | 格式选择依据 | 示例 |
|------|------------|------|
| **存储层** | 好不好增删改查统计 | JSON / 数据库表 |
| **提示词层** | 模型读不读得懂 | 自然语言描述 |
| **输出层** | 代码解析方不方便 | JSON Schema |

### 4.6 完整实例：运动计划查询的全流程

**场景**：用户问"今天什么时间做什么运动？"

**第一步：存储层（JSON）**

```json
{
  "user": {
    "name": "张阿姨",
    "age": 55,
    "exercisePlan": [
      { "weekday": [1, 3, 5], "time": "07:00", "type": "快走", "duration": 40 },
      { "weekday": [2, 4],    "time": "18:00", "type": "八段锦", "duration": 20 }
    ]
  },
  "today": "2026-04-13",   // 今天是周一，weekday=1
  "exerciseLog": []        // 今天还没记录运动
}
```

**第二步：检索后拼成提示词（自然语言）**

```
今天是2026年4月13日（周一）。

用户信息：张阿姨，55岁

运动计划：
- 周一、周三、周五早上7点快走40分钟
- 周二、周四下午6点八段锦20分钟

今天已记录的运动：无
```

**第三步：发给大模型（提示词）**

```
你是一个运动健康助手。今天是2026年4月13日（周一）。

用户：张阿姨，55岁

运动计划：
- 周一、周三、周五早上7点快走40分钟
- 周二、周四下午6点八段锦20分钟

今天已记录的运动：无

用户问："今天什么时间做什么运动？"

请根据运动计划回答，简洁友好。
```

**第四步：模型输出（JSON）**

```json
{
  "answer": "今天是周一，您计划早上7点快走40分钟。加油！💪"
}
```

**第五步：代码解析输出，展示给用户**

```
今天是周一，您计划早上7点快走40分钟。加油！💪
```

### 4.7 存储格式设计：结构化的好处

虽然存储什么格式都可以，但**结构化存储（JSON/数据库）**有这些优势：

| 能力 | 结构化存储 | 纯文本存储 |
|------|-----------|-----------|
| 查"今天周几该运动" | ✅ 一行代码 | ❌ 靠正则匹配 |
| 统计"本周运动几次" | ✅ 数组计数 | ❌ 靠AI提取 |
| 判断"目标完成了吗" | ✅ `current >= target` | ❌ 靠AI理解 |
| 主动提醒 | ✅ 代码可判断 | ❌ 必须等用户问 |

**结论**：
- **存储用JSON** → 应用好处理
- **提示词用文字** → 大模型好理解
- **输出用JSON** → 应用好解析 |

## 五、关键模块设计

### 5.1 第一层：快速匹配

**目标**：用正则匹配简单、无歧义的消息，**不调用大模型，零成本**

#### 5.1.1 完整的正则匹配规则（供参考）

以下是50+女性运动健康助手可以使用的正则匹配规则，按类型分组：

---

**类型一：寒暄问候**（用户打招呼，直接回应）

```javascript
// 寒暄问候规则
const GREETING_PATTERNS = [
  // 基础问候
  /^你好$/, /^您好$/, /^嗨$/, /^hi$/i, /^hello$/i,
  
  // 带时间问候
  /^早上好$/, /^早安$/, /^上午好$/,
  /^中午好$/, /^午安$/,
  /^下午好$/, /^傍晚好$/,
  /^晚上好$/, /^晚安$/,
  
  // 带语气词
  /^你好啊$/, /^您好啊$/, /^嗨呀$/,
  /^你好呀$/, /^您好呀$/,
  
  // 叠词/口语
  /^早$/, /^早早$/, /^哈喽$/i, /^哈罗$/i,
];
// 匹配后直接返回："您好！有什么可以帮您的吗？"
```

---

**类型二：感谢确认**（用户表示感谢或确认）

```javascript
// 感谢规则
const THANKS_PATTERNS = [
  // 基础感谢
  /^谢谢$/, /^感谢$/, /^谢了$/, /^谢啦$/,
  /^多谢$/, /^感谢谢$/,
  
  // 英文
  /^thx$/i, /^thanks$/i, /^thank you$/i,
  
  // 加强语气
  /^非常感谢$/, /^太感谢了$/, /^太谢谢了$/,
  /^谢谢你$/, /^感谢您$/,
];
// 匹配后直接返回："不客气！有需要随时找我～"

// 确认规则
const CONFIRM_PATTERNS = [
```

**讨论：简单指令要不要放第一层？**

| 观点 | 理由 |
|------|------|
| **放第一层** | 用户意图明确，不需要AI判断；省钱省时 |
| **不放第一层** | 后续需要提取参数（如"记录运动"需要问"什么运动"），正则无法处理 |

**建议**：简单指令**不放第一层**，让AI处理，因为：
- "记录运动" → AI识别后 → 询问"您做了什么运动？"
- "查看统计" → AI识别后 → 调用统计模块

**但可以放第一层的简单指令**（不需要后续参数）：

```javascript
// 可以放第一层的简单指令（不需要后续参数）
const SAFE_COMMAND_PATTERNS = [
```

```javascript
  // 帮助
  { patterns: [/^帮助$/, /^help$/i, /^怎么用$/, /^你能做什么$/],
    response: "我可以帮您：记录运动、记录饮食、查看统计、管理健康档案。您可以直接告诉我，比如'我今天散步了30分钟'。" },

  { patterns: [/^我的健康$/, /^健康档案$/, /^个人档案$/],
    intent: "health_query",  // 有意图，需要执行
    direct: true },
  { patterns: [/^我的统计$/, /^看统计$/, /^本周统计$/],
    intent: "stats_query",
    direct: true },

// 匹配函数
function fastMatch(message) {
  const text = message.trim();
  
  // 长度检查（超过15字通常不是简单对话）
  if (text.length > 15) return null;
  
  for (const rule of FAST_PATTERNS) {
    if (rule.patterns.some(p => p.test(text))) {
      // 有直接响应的，返回响应
      if (rule.response) {
        return { matched: true, response: rule.response };
      }
      // 有意图的，返回意图（交给后续处理）
      if (rule.intent) {
        return { matched: true, intent: rule.intent, direct: rule.direct };
      }
    }
  }
  
  return null;  // 未匹配，交给第二层
}
```

#### 5.1.3 匹配条件

不是所有消息都适合第一层匹配。需要满足：

| 条件 | 说明 | 示例 |
|------|------|------|
| **长度限制** | 不超过15个字 | "你好"✅ "我今天散步了30分钟感觉挺好的"❌ |
| **无歧义** | 只有一种理解 | "你好"只有问候一种理解 ✅ |
| **不需要参数** | 不需要提取额外信息 | "我的统计"不需要参数 ✅ "记录运动"需要问什么运动 ❌ |

**边界情况处理**：

```
用户说："你好啊今天天气真好"

问题：超过15字，但包含"你好"

处理：不匹配第一层，交给AI判断
      → AI可能判断为"寒暄+闲聊"
      → 回应问候，同时回应天气话题
```

---

#### 5.1.4 设计决策总结

| 问题 | 决策 | 原因 |
|------|------|------|
| "你好"放第一层吗？ | ✅ 放 | 意图明确，无歧义 |
| "记录运动"放第一层吗？ | ❌ 不放 | 需要后续问"什么运动" |
| "我的统计"放第一层吗？ | ✅ 放 | 意图明确，不需要参数 |
| 正则写多长？ | 精确匹配 | 避免误匹配，宁可漏也不要错 |

### 5.2 第二层：AI意图识别

**核心任务**：调用大模型判断用户意图，提取关键信息

#### 5.2.1 为什么需要提示词工程？

大模型不会自动理解你的业务。你需要告诉它：

| 你要告诉它的 | 为什么 |
|-------------|--------|
| 有哪些意图可选 | 否则它会乱猜 |
| 每个意图的定义 | 否则它会混淆 |
| 输出格式要求 | 否则返回的格式你没法解析 |
| 用户画像信息 | 否则它不知道用户背景 |

**提示词设计原则**：

1. **明确**：不要模糊，说清楚要什么
2. **结构化**：用JSON输出，方便程序解析
3. **约束**：限制输出范围，防止跑偏
4. **示例**：给几个例子，让模型理解

### 5.3 第三层：安全检查（代码实现）

#### 5.3.1 为什么需要第三层？

> **第二层 AI 识别 ≠ 第三层安全检查**
>
> 很多同学把 AI 输出的 `safetyCheck` 字段当成"第三层安全检查"，这是混淆了概念。
>
> - **第二层（AI 识别）**：AI 判断"这个消息有没有风险"，给出 flag
> - **第三层（代码安全检查）**：**代码强制执行**安全规则，不管 AI 说什么
>
> 为什么必须用代码执行，不能只靠 AI？
>
> | 问题 | AI 的局限 | 代码的优势 |
> |------|-----------|-----------|
> | AI 可能漏检 | 模型幻觉、Prompt 攻击 | 规则固定，100% 执行 |
> | AI 可能误判 | 不同模型判断不一致 | 同一规则，结果确定 |
> | 医疗责任 | AI 不能承担法律责任 | 系统可以声明"非医疗建议" |
>
> **结论**：AI 输出仅供参考，**代码必须独立执行安全规则**。

---

#### 5.3.2 安全规则体系

**三层安全规则**：

```
┌─────────────────────────────────────────────────────┐
│                    绝对禁止                         │
│  （任何情况下都不能做的事，发现即拦截）                  │
│  - 推荐禁忌运动给有相关疾病的患者                       │
│  - 胸痛/胸闷/呼吸困难等症状出现时建议继续运动            │
│  - 给出任何医疗诊断或治疗建议                           │
└─────────────────────────────────────────────────────┘
          ↓ 绝对禁止之外，继续检查 ↓
┌─────────────────────────────────────────────────────┐
│                    条件警告                          │
│  （满足条件时需要提示，但不是直接拦截）                  │
│  - 饮酒后运动 → 提示等待代谢后再运动                   │
│  - 高血压患者剧烈运动 → 提示温和运动                   │
│  - 老年人开始高强度运动 → 提示先咨询医生                │
└─────────────────────────────────────────────────────┘
          ↓ 警告之外，继续检查 ↓
┌─────────────────────────────────────────────────────┐
│                    建议升级                          │
│  （用户描述的症状可能需要医疗关注）                      │
│  - 头晕、关节疼痛持续 → 建议就医检查                   │
│  - 疑似疾病加重 → 建议咨询医生                        │
└─────────────────────────────────────────────────────┘
```

```javascript
// 第三部分：安全检查主函数
// ─────────────────────────────────────────────────────────

/**
 * 第三层安全检查
 *
 * @param {Object} intentResult - 第二层 AI 识别的结果
 * @param {string} intentResult.intent - 意图类型
 * @param {string} intentResult.subIntent - 子意图
 * @param {string} intentResult.extractedData - 提取的数据
 * @param {string} intentResult.safetyCheck - AI 返回的安全检查结果
 * @param {Object} userProfile - 用户画像
 * @param {string} userMessage - 原始用户消息
 * @returns {Object} 检查结果 { pass: boolean, response: string|null, level: string }
 */
function safetyCheck(intentResult, userProfile, userMessage) {
  const { intent, subIntent, extractedData, safetyCheck: aiSafety } = intentResult;
  const { health } = userProfile;

  // ── Step 1：绝对禁止检查（最高优先级）─────────────────
  const absoluteResult = checkAbsoluteForbidden(userMessage, userProfile);
  if (absoluteResult.blocked) {
    return {
      pass: false,
      response: SAFETY_RESPONSES.absoluteBlock(userMessage, absoluteResult.reason),
      level: 'BLOCK',           // 阻止执行
      reason: absoluteResult.reason
    };
  }

  // ── Step 2：推荐类意图 → 检查禁忌运动 ──────────────────
  if (['exercise', 'diet'].includes(intent) &&
      ['recommend', 'suggest'].includes(subIntent)) {
    const contraindicationResult = checkContraindications(
      extractedData,
      health,
      userMessage
    );
    if (contraindicationResult.blocked) {
      return {
        pass: false,
        response: SAFETY_RESPONSES.contraindicatedExercise(
          contraindicationResult.exercise,
          contraindicationResult.condition,
          contraindicationResult.alternatives
        ),
        level: 'BLOCK',
        reason: contraindicationResult.reason
      };
    }
    if (contraindicationResult.warned) {
      return {
        pass: true,
        response: SAFETY_RESPONSES.warning(contraindicationResult.warning),
        level: 'WARN',
        reason: contraindicationResult.reason
      };
    }
  }

  // ── Step 3：运动记录类 → 检查是否有不适症状 ──────────
  if (intent === 'exercise' && subIntent === 'log') {
    const symptomResult = checkSymptomsInLog(extractedData, userMessage);
    if (symptomResult.urgent) {
      return {
        pass: true,              // 允许记录，但不建议继续
        response: SAFETY_RESPONSES.urgentCare(symptomResult.symptoms),
        level: 'URGENT',
        reason: symptomResult.reason
      };
    }
    if (symptomResult.warned) {
      return {
        pass: true,
        response: SAFETY_RESPONSES.warning(symptomResult.warning),
        level: 'WARN',
        reason: symptomResult.reason
      };
    }
  }

  // ── Step 4：AI 安全检查补充（AI 返回了警告时）─────────
  if (aiSafety?.needReview === true && aiSafety?.warning) {
    // 如果 AI 返回了严重警告（以【必须拦截】开头），直接拦截
    if (aiSafety.warning.includes('【必须拦截】')) {
      return {
        pass: false,
        response: aiSafety.warning,
        level: 'BLOCK',
        reason: 'AI识别到严重安全风险'
      };
    }
    // 否则给出提示，但不阻止
    return {
      pass: true,
      response: SAFETY_RESPONSES.warning(aiSafety.warning),
      level: 'WARN',
      reason: 'AI识别到潜在风险'
    };
  }

  // ── Step 5：全部通过 ─────────────────────────────────
  return {
    pass: true,
    response: null,              // null = 无需额外响应
    level: 'PASS',
    reason: '安全检查通过'
  };
}

// ─────────────────────────────────────────────────────────
// 第四部分：各检查子函数
// ─────────────────────────────────────────────────────────

/**
 * 检查绝对禁止词
 */
function checkAbsoluteForbidden(message, userProfile) {
  const { absoluteForbidden } = CONTRAINDICATION_TABLE;

  // 检查禁忌症状词
  for (const symptom of absoluteForbidden.symptoms) {
    if (message.includes(symptom)) {
      return {
        blocked: true,
        reason: `您提到了"${symptom}"，这可能提示需要立即就医，不适合继续运动。`
      };
    }
  }

  // 检查禁忌行为
  for (const behavior of absoluteForbidden.behaviors) {
    if (message.includes(behavior)) {
      return {
        blocked: true,
        reason: `"${behavior}"存在安全隐患，不建议执行。`
      };
    }
  }

  return { blocked: false };
}

/**
 * 检查禁忌运动
 */
function checkContraindications(extractedData, userHealth, userMessage) {
  const { contraindications } = CONTRAINDICATION_TABLE;
  const userConditions = userHealth?.conditions || [];

  // 提取用户想做的运动类型
  const requestedExercise = extractedData?.exercise ||
                            extractedData?.action ||
                            extractedData?.goal || '';

  // 逐个检查用户的每种疾病
  for (const condition of userConditions) {
    const rule = contraindications[condition];
    if (!rule) continue;

    // 检查是否请求了禁忌运动
    for (const forbidden of rule.forbidden) {
      if (requestedExercise.includes(forbidden) ||
          userMessage.includes(forbidden)) {
        return {
          blocked: true,
          exercise: forbidden,
          condition: condition,
          reason: `${condition}患者不宜${forbidden}`,
          alternatives: getSafeAlternatives(condition)
        };
      }
    }

    // 检查警告级别
    for (const warned of rule.warn) {
      if (requestedExercise.includes(warned) ||
          userMessage.includes(warned)) {
        return {
          blocked: false,
          warned: true,
          exercise: warned,
          condition: condition,
          warning: `${condition}患者${warned}需谨慎，建议选择更温和的运动`,
          reason: `${condition}患者${warned}需谨慎`
        };
      }
    }
  }

  // 额外检查：马拉松等高强度运动的通用拦截
  const highIntensityPatterns = ['马拉松', 'HIIT', '高强度间歇'];
  for (const pattern of highIntensityPatterns) {
    if (userMessage.includes(pattern)) {
      // 检查用户健康状况是否适合高强度
      const riskyConditions = ['高血压', '心脏病', '骨质疏松', '膝关节', '腰椎'];
      const hasRisk = userConditions.some(c => riskyConditions.some(r => c.includes(r)));
      if (hasRisk) {
        return {
          blocked: true,
          exercise: pattern,
          condition: userConditions.find(c => riskyConditions.some(r => c.includes(r))),
          reason: `高强度运动（${pattern}）可能对您的健康状况有风险`,
          alternatives: getSafeAlternatives(userConditions[0])
        };
      }
    }
  }

  return { blocked: false, warned: false };
}

/**
 * 检查运动记录中的症状
 */
function checkSymptomsInLog(extractedData, userMessage) {
  const symptoms = extractedData?.symptom || '';

  for (const urgent of CONTRAINDICATION_TABLE.urgentCare) {
    if (userMessage.includes(urgent) || symptoms.includes(urgent)) {
      return {
        urgent: true,
        symptoms: urgent,
        reason: `用户报告了需要关注的症状：${urgent}`
      };
    }
  }

  // 检查一般不适关键词
  const generalSymptoms = ['有点累', '比较累', '不舒服', '酸痛'];
  for (const symptom of generalSymptoms) {
    if (userMessage.includes(symptom)) {
      return {
        warned: true,
        warning: `您提到了${symptom}，运动后注意休息，如有加重请就医。`,
        reason: `用户报告了一般不适症状：${symptom}`
      };
    }
  }

  return { urgent: false, warned: false };
}

/**
 * 获取安全替代运动
 */
function getSafeAlternatives(condition) {
  const alternatives = {
    '高血压': ['散步', '太极拳', '八段锦', '慢走', '轻度广场舞'],
    '骨质疏松': ['游泳', '骑自行车', '太极拳', '轻度步行'],
    '膝关节': ['游泳', '太极拳', '卧式健身车', '上肢训练'],
    '腰椎': ['游泳', '散步', '太极拳（避免弯腰）'],
    '心脏病': ['步行', '轻度太极拳', '呼吸训练'],
    '糖尿病': ['散步', '太极拳', '游泳', '轻度有氧运动']
  };
  return alternatives[condition] || ['散步', '太极拳'];
}
```

```javascript
async function handleMessage(userMessage, userProfile) {
  // ── 第一层：快速匹配（正则，零成本）────────────────
  const fastResult = fastMatch(userMessage);
  if (fastResult) {
    if (fastResult.response) return fastResult.response;    // 直接返回
    if (fastResult.intent) {
      // 需要执行的简单指令
      return await executeIntent(fastResult.intent, userProfile);
    }
  }

  // ── 第二层：AI 意图识别 ────────────────────────────
  const intentResult = await recognizeIntent(userMessage, userProfile);

  // ── 第三层：代码安全检查（不管 AI 返回什么，这里必执行）─
  const safetyResult = safetyCheck(intentResult, userProfile, userMessage);

  // 根据安全检查结果决定如何处理
  if (!safetyResult.pass) {
    // 拦截：返回安全提示，不执行原意图
    return safetyResult.response;
  }

  if (safetyResult.response) {
    // 警告：打印提示，继续执行
    console.warn(`[安全警告] ${safetyResult.reason}`);
    const actionResult = await executeIntent(intentResult, userProfile);
    return `${safetyResult.response}\n\n${actionResult}`;
  }

  // ── 第四层：执行意图 ───────────────────────────────
  return await executeIntent(intentResult, userProfile);
}
```

## 七、为什么这样设计？—— 常见问题解答

### Q1：为什么不全部用AI判断？

**A**：成本和速度。

| 方式 | 单次成本 | 单次延迟 | 适用场景 |
|------|---------|---------|---------|
| 正则匹配 | 0 | <5ms | 寒暄、简单确认 |
| AI判断 | >0.001元 | 500ms-1s | 复杂意图 |

**结论**：能用简单方法解决的问题，不要浪费AI资源。

### Q2：正则匹配会不会太死板？

**A**：会。但第一层只匹配"无歧义的简单消息"，不影响复杂场景的处理。

**原则**：第一层只处理"所有人都一样"的场景。

```
❌ 第一层不应该匹配：
"我今天散步了"  → 不同用户意图不同

✅ 第一层可以匹配：
"你好"         → 所有人都一样，问候而已
"好的"         → 所有人都一样，确认而已
```

### Q3：AI识别结果不可靠怎么办？

**A**：设置置信度阈值。

```javascript
const CONFIDENCE_THRESHOLD = 0.7;

if (result.confidence < 0.7) {
  // 置信度太低，不确定用户意图
  // → 降级为普通对话，不执行具体功能
  return "我没太明白您的意思，能再说一次吗？"
}
```

### Q4：意图识别错误怎么办？

**A**：有两条防线。

```
第一道：置信度检查
  - AI返回置信度 < 0.7 → 不执行具体功能

第二道：安全检查
  - 即使意图识别对了，安全检查也会拦截危险操作
```

## 九、总结

### 本节要点

1. **意图识别的目的**：让助手听懂用户想做什么
2. **分层处理**：简单问题用正则，复杂问题用AI
3. **安全第一**：健康领域必须有安全检查
4. **置信度机制**：不确定时降级处理

### 架构图回顾

```
用户消息
   │
   ├── 快速匹配（正则）→ 问候/确认/告别
   │
   └── AI识别意图
            │
            ├── 置信度 < 0.7 → 降级为普通对话
            │
            └── 置信度 ≥ 0.7
                      │
                      ├── 安全检查 → 拦截危险操作
                      │
                      └── 执行对应能力
```
