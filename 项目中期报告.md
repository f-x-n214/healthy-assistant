- **项目名称**：银发健康助手
- **目标用户**：65岁以上独居老人（父母端）及其子女（子女端）

### 3.2 关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 意图识别架构 | 三层：正则→AI→安全检查 | 第一层零成本处理简单消息节省API调用，第二层处理复杂语义，第三层安全兜底防止危险输出 |
| 前端框架 | 纯JS（无React/Vue） | 适老化项目页面少、交互简单，组件化收益有限；纯JS减少构建复杂度，调试直观 |
| 数据存储 | JSON文件 + ChromaDB | JSON文件简单可靠易调试，ChromaDB支持语义检索实现长期记忆的"回忆"能力 |
| LLM模型 | 阿里云Qwen-Plus (DashScope) | 智谱GLM免费额度用完后切换，Qwen-Plus兼容OpenAI格式，性价比高 |
| 告警通知 | localStorage + 后端API双写 | localStorage即时可用无延迟，后端API支持跨设备同步（子女端不同设备获取告警） |
| 提醒轮询 | 前端30秒定时器 | 简单可靠，无需WebSocket基础设施；后续可升级为SSE |
| 情绪关怀 | 关键词分类+随机回复池 | 按情绪类型（伤心/害怕/焦虑/孤独等）给出差异化温暖回应，随机选择避免机械重复 |
| 适老化设计 | CSS变量体系+全局大字号 | --font-base:20px起步，所有控件min-height:48px，高对比度配色 |

## 四、与Agent协作总结

### 4.1 协同方式

- **Agent**：华为云CodeArts（码道）代码智能体
- **使用方式**：在CodeArts IDE中通过自然语言指令与Agent协作开发。我描述需求和验收标准 → Agent实现代码 → 我逐项验证 → 发现问题后引导Agent修复 → 迭代优化
- **使用频率**：每日开发全程使用，平均每天5-8轮对话交互

### 4.2 协同案例

#### 案例1：告警检测链路的四层根因分析与修复

**我提出的需求**："为什么用户说'吐了好多血'却没有检测到异常发出警告？子女端也没有弹出异常提醒？"

**Agent排查结果**：发现了**四个层面**的根因，我逐条审查验证：

| 层次 | 根因 | 我的验证 |
|------|------|---------|
| 第1层 | 出血关键词`"吐血"`用`includes`匹配，口语"吐了好多血"中间隔了字，匹配失败 | ✅ 确认，需要正则模式匹配 |
| 第2层 | index.html缺少`alertContainer`元素，告警通知无处渲染 | ✅ 确认，HTML结构缺失 |
| 第3层 | 子女端横幅只统计URGENT级别，WARN级别告警不显示 | ✅ 确认，逻辑不完整 |
| 第4层 | 告警只存localStorage，无后端存储，不同设备的子女端无法获取 | ✅ 确认，跨设备通知断裂 |

**我的决策**：要求Agent修复全部四层问题，不接受只修表面。Agent修复后，我又发现点击告警横幅不跳转——因为`type="module"`中函数不在全局作用域，`onclick`调用不到。再次要求修复，Agent将函数挂载到`window`解决。

**关键体会**：Agent擅长深度排查根因链，但需要人指定排查方向和验收标准。四层问题一次修完比修四次效率高得多。

#### 案例2：提醒删除序号错位Bug的迭代修复

**问题现象**：用户说"删除第5条提醒"（对应已关闭列表中"其他 每天10:23"），但实际删除了第4条（"其他 每天10:15"）。

**Agent首次修复**：采用"智能判断分区"方案——根据序号大小推断用户指的是哪个分区。

**我的验证与否决**：我发现该方案仍有问题，且语义不够直观。我明确提出精确行为规格：

> "开启和已关闭各自独立编号。说'删除第N条'则两个分区的第N条都删除。没有的就跳过。"

**Agent按规格重写**：分区独立编号+按操作类型决定目标列表，修复成功。

**关键体会**：模糊指令导致Agent方案偏移。当我给出精确的行为规格后，一次修复到位。这是"精确指令价值"的最直接例证。

### 4.3 协同经验

1. **精确指令的价值**：说"关闭第N条的逻辑有问题"Agent会猜测意图，可能修偏；说"开启和已关闭各自独立编号，未指定分区时两个分区的第N条都操作"则一次修复到位。**精确度直接决定迭代轮数**，模糊指令的代价是2-3轮额外迭代。

2. **分步验证的重要性**：Agent一次修改可能影响其他页面。例如Agent修改了全局`.main`的CSS导致健康档案页面布局崩溃。我养成习惯：每次修改后立即检查其他页面是否受影响，发现后要求Agent用作用域限定（如`.main--chat`）而非全局修改。**局部修改必须用作用域隔离，绝不能改全局样式**。

3. **根因驱动而非现象驱动**：Agent倾向于修表面现象（如给情绪关怀加个前缀字符串），我要求追到根因（回复机械化的根因是回复池太单一+情绪类型没有细分），从根本上解决才不会反复出问题。**修现象会复发，修根因才根治**。

### 5.1 遇到的问题

| # | 问题 | 原因分析 | 解决过程 |
|---|------|---------|---------|
| 1 | API 429限流 | 智谱GLM-4-7模型免费额度用完 | 先在模型平台关闭安全体验模式暂时恢复；后续切换到阿里云Qwen-Plus |

| 4 | 告警检测四层断裂 | 关键词匹配不支持口语化+UI容器缺失+横幅级别不全+无后端存储 | 四层根因逐一修复：正则模式+添加容器+横幅显示WARN+后端API双写 |
| 5 | 删除提醒序号错位 | 全局编号与用户看到的分区编号不一致 | 重写为分区独立编号+按操作类型(关闭/开启/删除)决定目标列表 |
| 6 | 全局CSS修改影响其他页面 | `.main`样式被所有页面共用，改为flex布局后健康档案页面崩溃 | 用`.main--chat`作用域限定，仅影响首页 |

### 5.2 计划调整

| 调整项 | 原计划 | 实际调整 | 原因 |
|--------|--------|---------|------|
| 告警通知机制 | 纯localStorage | localStorage + 后端API双写 | 需支持跨设备同步，子女端在不同浏览器/设备也能获取告警 |
| 子女端告警获取 | 依赖localStorage | 优先从后端API加载，localStorage降级 | localStorage仅同浏览器生效，后端API才是可靠数据源 |
| 情绪关怀 | 固定前缀"我理解你的感受" | 8种情绪分类×多句随机回复池 | 老年人需要更有温度的陪伴，固定回复显得敷衍 |
| 对话历史 | 仅存内存MEMORY.history | 后端持久化+页面恢复 | 刷新不丢失对话，恢复最近50条历史 |
| LLM模型 | 智谱GLM-4-7 | 阿里云Qwen-Plus (DashScope) | 智谱免费额度用完，Qwen-Plus兼容OpenAI格式 |
| 首页布局 | 标题+小贴士+快捷按钮+聊天区 | 顶部栏(品牌+快捷按钮)+聊天区占满 | 聊天区从约50%提升到约85%，更适合老年人对话 |
