## 1 项目概述

### 1.1 选题动机

随着我国老龄化进程加速，65岁以上人口已超过2亿，其中大量为独居、慢病群体。这些老年人在日常健康管理中面临诸多困难：用药记录靠纸笔容易遗忘、健康数据分散难以追踪、子女远程照护缺乏信息通道、紧急情况求助不便。

与此同时，AI大模型技术快速发展，大模型对话框（如ChatGPT、通义千问等）已能完成许多任务。然而，直接使用对话框存在根本性局限：

| 对话框的优势 | 对话框的劣势 |
|-------------|-------------|
| 直接、通用，无需安装 | 每次都从零开始，无法持久记忆 |
| 随时可用，免费或低成本 | 只能生成文本，不能执行操作 |
| 模型持续更新 | 被动响应，不会主动提醒 |

**核心洞察**：对话框是"问"，应用是"做"。用户愿意多一步选择独立应用，只因为它能帮他"做"事，而不仅仅帮他"说"话。

以健康管理为例，对话框可以说"运动对身体好"，但不能：
- 把用户今天的运动记录**存下来**
- 到了该吃药的时间**主动提醒**
- 检测到血压异常时**自动告警**
- 让远在异地的子女**查看父母数据**

这些"做"的能力，正是独立应用存在的价值。因此，本项目选择开发一个**超越对话框的智能健康管理应用**，面向65+中老年人，让AI真正成为能"做事"的健康伙伴。

### 1.2 项目目标与范围

本项目旨在开发一个面向65岁以上中老年人（独居、慢病群体）及其子女（照护者）的AI健康管理助手——**银发健康助手**。

**核心目标**：

1. **用最简单的对话帮助记录用药与健康数据**：用户只需说"我今天散步了30分钟"，系统自动识别意图、提取信息、存储数据
2. **提供安全健康科普咨询**：在医疗安全底线之上，基于用户画像提供个性化健康建议
3. **支持子女在授权下查看父母数据**：通过家庭绑定机制实现远程照护信息通道
4. **银发适老化设计**：大字体、高对比度、语音交互，降低使用门槛

**项目范围**：

| 范围内 | 范围外 |
|--------|--------|
| 智能对话+意图识别+安全检查 | 医疗诊断与处方 |
| 用药/血压/血糖/体重/运动/饮食记录 | 与医院HIS系统对接 |
| 定时提醒+依从性统计 | 可穿戴设备数据采集 |
| 家庭绑定+子女远程查看 | 社区养老平台集成 |
| 天气健康建议 | 多语言/方言支持 |
| 紧急求助（拨打120/联系人） | 120自动报警 |

### 1.3 技术选型概述

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| **前端** | 原生HTML5+CSS3+JS (ES Module) | - | 银发用户设备性能有限，原生开发零依赖、加载快 |
| **图表** | Chart.js | CDN引入 | 轻量级，健康数据可视化 |
| **语音** | Web Speech API + Speech Synthesis | 浏览器原生 | 零依赖，支持zh-CN普通话识别与播报 |
| **后端** | Python Flask | >=3.0 | 轻量级，适合中小规模API服务 |
| **向量库** | ChromaDB | >=0.4.22 | 语义检索，HNSW算法，cosine距离 |
| **关系库** | SQLite3 | Python内置 | 家庭绑定关系存储，嵌入式无需部署 |
| **数据存储** | JSON文件 | - | 健康记录按月分文件，简单可靠 |
| **大模型** | 阿里云通义千问 | qwen-plus | 国产模型，OpenAI兼容接口，成本低 |

**选型决策说明**：

1. **前端不用Vue/React**：银发用户手机性能有限，原生JS零框架依赖，首屏加载<1秒
2. **后端不用Spring Boot**：项目数据模型简单，Flask足够；Python生态对ChromaDB支持更好
3. **JSON文件而非MySQL**：健康记录按月分文件（如`default_bp_2026-06.json`），查询单月只需读一个文件，部署零配置
4. **SQLite仅用于家庭绑定**：绑定关系需要事务和JOIN查询，SQLite嵌入式无需额外部署

### 3.2 三级意图识别设计

意图识别是本系统的核心智能特性，采用分层处理架构：

```
用户消息
   │
   ▼
第一层：正则词库快速匹配（firstLayer.js + fastReplies.js）
   │  <5ms，零成本
   ├── 匹配成功 → 直接返回（问候/确认/告别等）
   │
   └── 未匹配 ↓
        │
        ▼
第二层：AI意图识别（app.js callLLM()，调用通义千问qwen-plus）
   │  500ms-1.5s
   │  返回：意图类型 + 置信度 + 提取数据
   │
   ├── 置信度 < 0.7 → 降级为普通对话
   │
   └── 置信度 ≥ 0.7 ↓
            │
            ▼
第三层：安全检查（safetyRules.js，代码强制执行）
   │
   ├── BLOCK → 拦截，返回安全提示
   ├── URGENT → 执行 + 紧急就医提示
   ├── WARN → 执行 + 风险警告
   └── PASS → 正常执行
```

**设计决策**：

| 决策 | 为什么 | 类比 |
|------|--------|------|
| 第一层用正则不用AI | "你好"不需要AI判断 | 问路不需要查地图 |
| 第二层才用AI | 复杂语义AI才能准确判断 | 复杂问题才需要专家 |
| 第三层用代码不用AI | AI可能漏检误判，健康领域容不得错误 | 医疗设备有安全阀 |

### 3.3 记忆系统设计

对话框每次从零开始，应用必须记住一切。本系统设计三层记忆架构：

```
长期记忆（持久化）
├── 用户画像（profile.json）：姓名/年龄/慢性病/过敏/用药/行为阶段
├── 核心记忆（memory.json）：里程碑/重要决定/深层动机
└── 健康记录（按月分文件）：med/bp/bs/wt/diet_YYYY-MM.json

语义记忆（向量检索）
└── ChromaDB（HNSW, cosine）：memory_{user_id} collection

短期记忆（会话级）
├── 对话历史（chat_history.json，启动时恢复）
└── 分级TTL缓存（profile:会话级 / records:30秒 / statistics:10分钟）
```

### 3.4 提醒引擎设计

```
定时触发 → 用户通知 → 确认/跳过 → 10分钟重提醒 → 漏服计数 → 通知子女
```

提醒引擎实现"持续运行"能力：不需要用户主动询问，系统主动推送提醒，形成用药依从性闭环。

### 3.5 家庭照护设计

```
父母生成邀请码 → 子女输入邀请码 → 父母确认绑定 → 子女选择权限 → 子女查看数据
```

权限类型：medication / blood_pressure / blood_sugar / weight / emergency，实现细粒度授权。

### 3.6 银发适老化设计

| 原则 | 实现措施 |
|------|---------|
| 看得清 | 4档字体（基准24-30px）、高对比度、大间距 |
| 点得着 | 大按钮（≥48px触摸目标）、底部4Tab导航 |
| 听得懂 | 语音播报关键信息、简洁文案、避免技术术语 |

## 4 核心功能实现

### 4.1 三级意图识别

#### 4.1.1 第一层：正则词库快速匹配

**实现文件**：`frontend/intent/firstLayer.js` + `frontend/intent/fastReplies.js`

`firstLayer.js` 定义8大词库类型，约60个关键词：

| 词库类型 | 示例关键词 | 返回行为 |
|---------|-----------|---------|
| 寒暄问候 | 你好、您好、早上好、晚安 | 直接返回问候语 |
| 感谢确认 | 谢谢、好的、可以、没问题 | 直接返回确认语 |
| 告别结束 | 再见、拜拜、明天见 | 直接返回告别语 |
| 使用帮助 | 帮助、怎么用、你能做什么 | 返回功能介绍 |
| 用药提醒开关 | 提醒我吃药、关闭提醒 | 路由到提醒模块 |
| 强格式记录 | 血压130/80、血糖5.6 | 直接提取数值存储 |
| 子女查看数据 | 我妈的血压、爸爸的用药 | 路由到子女查看 |
| 情绪关怀 | 心情不好、有点累 | 路由到情绪关怀 |

**核心代码**（出处：`frontend/intent/firstLayer.js`，`runFirstLayer()`函数）：

```javascript
export function runFirstLayer(text, profile) {
  if (!text || typeof text !== 'string') return null;
  const t = text.trim();
  if (t.length > 18) return null;  // 超过18字交给第二层

  // 精确匹配（优先）
  for (const rule of EXACT_RULES) {
    if (t === rule.keyword) return { ...rule.result, _layer: 1, _type: 'exact' };
  }

  // 正则匹配
  for (const rule of PATTERN_RULES) {
    if (rule.pattern.test(t)) return { ...rule.result, _layer: 1, _type: 'pattern' };
  }

  // 包含匹配（最宽松）
  for (const rule of INCLUDE_RULES) {
    if (t.includes(rule.keyword)) return { ...rule.result, _layer: 1, _type: 'include' };
  }

  return null;  // 未匹配，交给第二层
}
```

#### 4.1.3 第三层：安全检查

**实现文件**：`frontend/intent/safetyRules.js`，`runSafetyCheck()` 函数

四级安全体系：

| 级别 | 行为 | 适用场景 | 示例 |
|------|------|---------|------|
| **BLOCK** | 拦截，不执行 | 自伤/自杀、危险用药、胸痛/呼吸困难 | 高血压患者推荐快跑 |
| **URGENT** | 执行+紧急提示 | 需要立即就医的症状 | 血压≥180/120 |
| **WARN** | 执行+警告提示 | 潜在风险行为 | 饮酒后运动、擅自调药 |
| **PASS** | 正常执行 | 普通记录/查询/科普 | 记录今天散步30分钟 |

**核心代码**（出处：`frontend/intent/safetyRules.js`，`runSafetyCheck()`函数）：

```javascript
export function runSafetyCheck(text, intentResult, profile) {
  // Step 1: 绝对禁止检查（最高优先级）
  for (const rule of BLOCK_RULES) {
    if (rule.pattern.test(text)) {
      return { pass: false, level: 'BLOCK', response: rule.response, reason: rule.reason };
    }
  }

  // Step 2: 紧急症状检查
  for (const rule of URGENT_RULES) {
    if (rule.pattern.test(text)) {
      return { pass: true, level: 'URGENT', response: rule.response, reason: rule.reason };
    }
  }

  // Step 3: 条件警告检查
  for (const rule of WARN_RULES) {
    if (rule.pattern.test(text) && rule.condition(profile)) {
      return { pass: true, level: 'WARN', response: rule.response, reason: rule.reason };
    }
  }

  // Step 4: 全部通过
  return { pass: true, level: 'PASS', response: null };
}
```

```javascript
export function extractProfileData(text) {
  const result = {};
  // 姓名提取
  const nameMatch = text.match(/我叫(.{2,4})|我是(.{2,4})/);
  if (nameMatch) result.name = nameMatch[1] || nameMatch[2];
  // 年龄提取
  const ageMatch = text.match(/(\d{2,3})\s*岁/);
  if (ageMatch) result.age = parseInt(ageMatch[1]);
  // 慢性病提取
  const diseases = ['高血压', '糖尿病', '骨质疏松', '冠心病', '高血脂'];
  for (const d of diseases) {
    if (text.includes(d)) result.chronicDiseases = [...(result.chronicDiseases || []), d];
  }
  return Object.keys(result).length > 0 ? result : null;
}
```

```python
def save_record(self, user_id, record_type, data):
    now = datetime.now()
    month_key = now.strftime("%Y-%m")
    filename = f"{user_id}_{record_type}_{month_key}.json"
    filepath = os.path.join(self.data_dir, filename)
    
    records = []
    if os.path.exists(filepath):
        with open(filepath, 'r', encoding='utf-8') as f:
            records = json.load(f)
    
    record = {**data, "id": f"{record_type}_{int(now.timestamp())}", "date": now.isoformat()}
    records.append(record)
    
    with open(filepath, 'w', encoding='utf-8') as f:
        json.dump(records, f, ensure_ascii=False, indent=2)
    
    self._add_to_chroma(user_id, str(record), {"type": record_type, "date": now.isoformat()})
    return record
```

```python
def check_and_fire(self, user_id):
    reminders = self.memory.get_reminders(user_id)
    now = datetime.now()
    fired = []
    for r in reminders:
        if not r.get("enabled", True): continue
        if self._should_fire(r, now):
            msg = self._build_message(r)
            self._record_trigger(user_id, r["id"])
            fired.append({"id": r["id"], "type": r["type"], "message": msg})
    return fired
```

```python
def validate_access(self, child_id, parent_id, permission_type):
    with sqlite3.connect(self.db_path) as conn:
        row = conn.execute("""
            SELECT fp.enabled FROM family_bonds fb
            JOIN family_permissions fp ON fp.bond_id = fb.id
            WHERE fb.child_id=? AND fb.parent_id=? AND fb.status='accepted'
            AND fp.permission_type=? AND fp.enabled=1
        """, (child_id, parent_id, permission_type)).fetchone()
    return row is not None
```

### 6.3 不足与展望

#### 6.3.1 当前不足

1. **API密钥安全**：通义千问API密钥硬编码在前端`app.js`中，生产环境需移至后端代理
2. **并发性能**：Flask默认单线程，多用户并发性能受限
3. **离线能力**：依赖后端和LLM API，网络断开时功能受限
4. **测试覆盖**：仅有`profileExtractor.test.js`和`memoryTest.js`，覆盖率不足
5. **数据迁移**：JSON文件存储不便于迁移和备份

#### 6.3.2 改进方向

1. **API密钥安全**：LLM调用移至后端代理
2. **数据库升级**：JSON迁移至SQLite/MySQL
3. **PWA离线支持**：Service Worker缓存关键资源
4. **方言支持**：粤语/四川话等，覆盖更多老年用户
5. **多Agent架构**：不同健康领域由专业Agent处理

### 6.4 核心结论

> **对话框是"问"，应用是"做"。**
> 
> 银发健康助手的核心价值不在于"能对话"，而在于"能做事"——能记住用户的健康数据、能主动提醒用药、能在危险时拦截、能让子女远程照护。这些是任何大模型对话框都无法提供的价值。
>
> 不是做一个"更好的对话框"，而是做一个"真正的智能伙伴"。
