# 项目中期报告 - 作业要求与评分标准

> 课程：智能应用项目开发综合实践
> 作业名称：项目中期报告
> 总分：100分

---

## 一、作业概述

### 1.1 作业目的

1. 检查项目按计划执行的实际进展
2. 验证核心功能是否已实现并可运行
3. 发现问题并及时调整，避免期末突击
4. 督促同学们持续推进项目，保持开发节奏
5. 总结与编程Agent协作的经验，提升协同开发能力

### 1.2 与第一次汇报的关系

| 对比项    | 第一次汇报（选题报告） | 中期报告         |
| ------ | ----------- | ------------ |
| **阶段** | 规划阶段        | 执行阶段         |
| **重点** | 做什么、为什么做    | 做了什么、做得怎样    |
| **产出** | 需求文档、设计文档   | 代码、运行效果、测试结果 |
| **验证** | 逻辑自洽即可      | 必须可运行、可演示    |
| **风险** | 方向偏差        | 进度滞后、技术困难    |

> **核心区别**：中期报告不是"写出来的"，而是"做出来的"。
> 
> 必须有真实运行效果，不接受纯文档、无实现汇报。

### 1.3 提交要求

| 项目       | 要求                 |
| -------- | ------------------ |
| **文档格式** | PDF格式              |
| **文档命名** | 学号-姓名-项目中期报告.pdf   |
| **现场演示** | 尽量发布到公网可访问或远程演示    |
| **演示视频** | 条件不允许公网部署则录3分钟内短视频 |
| **提交时间** | 学习通截止时间            |
| **提交方式** | 文档上传到课程平台          |

> **演示方式说明**：
> 
> - **优先**：将项目部署到公网（如Vercel、Render、华为云等），汇报时直接访问在线地址演示
> - **备选**：如果公网部署条件不允许，录制3分钟以内的演示短视频，汇报时播放视频
> - 两种方式都需要在文档中提供：**在线演示地址 或 视频链接**

## 二、文档结构要求

文档必须包含以下六大部分：

### 第一部分：项目进展总览

```markdown
## 一、项目进展总览

### 1.1 项目基本信息
（项目名称、目标用户、核心问题——简要回顾，不超过5行）

### 1.2 迭代计划执行情况
（对照选题报告的10-12周计划，标注每周完成状态）

### 1.3 总体完成度
（已完成功能占总功能的百分比，附说明）
```

### 第二部分：已完成功能详述

```markdown
## 二、已完成功能详述

### 2.1 已完成功能列表
（列出所有已实现的功能，每个功能附：实现周次、核心代码片段、运行截图）

### 2.2 核心功能演示
（至少3个核心功能的完整演示：用户输入→系统响应→运行截图/视频）

### 2.3 智能特性实现情况
（长期记忆、主动服务等核心特性的实现状态和效果）
```

### 第三部分：技术实现与架构

```markdown
## 三、技术实现与架构

### 3.1 实际技术架构
（实际采用的架构，与选题报告对比是否有调整）

### 3.2 关键技术决策
（开发过程中做出的关键技术决策及理由）

### 3.3 核心代码展示
（展示2-3段核心代码片段，每段不超过15行，附简要说明）
```

### 第四部分：与Agent协作总结

```markdown
## 四、与Agent协作总结

### 4.1 协同方式
（使用了哪个Agent、如何使用、使用频率）

### 4.2 协同案例
（描述1-2个典型协作场景：人提了什么需求、Agent如何实现、遇到了什么问题、如何解决）
（重点：不是Agent替你写代码，而是你与Agent协同完成——你做决策和验证，Agent做实现和迭代）

### 4.3 协同经验
（总结2-3条与Agent协同的经验教训，如：精确指令的价值、分步验证的重要性、方案迭代的过程等）
```

### 第五部分：问题与调整

```markdown
## 五、问题与调整

### 5.1 遇到的问题
（开发中遇到的技术问题、需求问题、协作问题）

### 5.2 计划调整
（与原计划相比的调整：提前完成、延后、取消、新增的功能）

### 5.3 风险评估更新
（当前风险状态，与选题报告的风险对比）
```

### 第六部分：后续计划

```markdown
## 六、后续计划

### 6.1 剩余功能规划
（未完成功能的实现计划，重新排优先级）

### 6.2 下阶段重点
（接下来2-3周的重点工作）

### 6.3 期末目标
（期末最终演示时计划达到的效果）
```

## 三、评分标准

### 总分：100分

| 评分项              | 分值      | 评分要点                         |
| ---------------- | ------- | ---------------------------- |
| **一、项目进展总览**     | **10分** | 计划执行情况真实；完成度自评有依据            |
| **二、已完成功能详述**    | **25分** | 功能列表完整；至少3个核心功能有演示；智能特性有实现效果 |
| **三、技术实现与架构**    | **15分** | 架构合理；技术决策有理由；代码组织清晰          |
| **四、与Agent协作总结** | **20分** | 协同方式明确；有典型协作案例（含问题与解决）；有经验总结 |
| **五、问题与调整**      | **15分** | 如实反映问题；调整合理有据                |
| **六、后续计划**       | **10分** | 未完成功能有计划；期末目标可验证             |
| **七、文档规范**       | **5分**  | PDF格式规范，截图清晰                 |

---

## 四、各评分项说明

### 4.1 项目进展总览（10分）

- **8-10分**：完整对照12周计划，每周标注完成状态，完成度自评有依据，真实不虚报
- **5-7分**：有计划执行情况，但部分周次标注不够明确或完成度依据不充分
- **1-4分**：计划执行情况描述不完整，或完成度明显虚高

### 4.2 已完成功能详述（25分）

> **这是中期报告最重要的部分。**

- **20-25分**：功能列表完整（附核心代码片段）；至少3个核心功能有完整演示（输入→响应→截图/视频）；长期记忆和主动服务都有实际效果
- **13-19分**：功能列表基本完整；有核心功能演示但不够完整；智能特性实现了至少一个
- **5-12分**：功能列表不完整；演示少于3个；智能特性只有框架无效果
- **0-4分**：缺少核心功能演示，或两个核心特性都未实现

**演示要求**：每个核心功能必须包含**用户输入 + 系统响应 + 截图或视频**，不接受纯文字描述。

**智能特性验收**：

| 特性   | 验收方法                               |
| ---- | ---------------------------------- |
| 长期记忆 | 第1天告诉助手信息，第2天新会话中助手能使用该信息（附两次会话截图） |
| 主动服务 | 助手能在设定时间主动推送消息（附推送截图和定时配置）         |

### 4.3 技术实现与架构（15分）

- **12-15分**：实际架构清晰合理（有架构图），与设计一致或合理调整有说明；至少2个关键技术决策有理由；目录结构清晰，核心代码展示恰当
- **8-11分**：架构基本合理；有技术决策但理由不够充分；有代码展示
- **1-7分**：架构不够清晰，缺少技术决策说明

### 4.4 与Agent协作总结（20分）

> **本项强调"协同"而非"代工"——不是Agent替你写代码，而是你与Agent协同完成。**

- **16-20分**：协同方式明确（哪个Agent、如何使用）；至少1个典型协作案例（描述了人提需求→Agent实现→发现问题→解决迭代的过程）；总结2-3条有价值的协同经验
- **10-15分**：有协同方式说明和案例，但案例描述不够深入，或经验总结较泛
- **5-9分**：只简单提及使用了Agent，缺少具体案例或经验
- **0-4分**：未提及与Agent协作，或描述为"Agent帮我写完了代码"（缺乏协同意识）

**协作案例参考维度**（不需要全部包含，选择有体会的写）：

- 方案演进：初始方案→发现问题→调整方案的过程
- 迭代节奏：分步实现→逐步验证的节奏把控
- 指令质量：精确指令与模糊指令的效果对比
- Bug发现：人发现Agent代码中的问题并引导修复的过程
- 角色认知：人对自身"决策者+验证者"角色的理解

### 4.5 问题与调整（15分）

- **12-15分**：如实反映至少3个实际问题（含问题描述和解决过程）；计划调整有明确理由，方向合理
- **8-11分**：反映了部分问题；有调整但部分理由不够充分
- **1-7分**：问题识别不充分，或声称"没有遇到问题"（不真实）

> **重要**：如实反映问题不会扣分，反而说明你在认真开发。声称"没有遇到问题"反而可疑。

### 4.6 后续计划（10分）

- **8-10分**：未完成功能有明确实现计划和优先级；期末目标具体可验证
- **4-7分**：有规划但不够具体
- **1-3分**：缺少后续规划或期末目标

### 4.7 文档规范（5分）

- **4-5分**：PDF格式规范，结构清晰，截图清晰
- **2-3分**：格式基本规范，有少量问题
- **1分**：格式不够规范

## 五、特别要求

### 5.1 必须有真实运行效果

- 核心功能必须有截图或视频，不接受纯文字描述
- **优先**：部署到公网，汇报时在线演示（推荐Vercel、Render、华为云等免费平台）
- **备选**：公网部署条件不允许时，录制3分钟内演示短视频

### 5.2 智能特性必须有效果

- **长期记忆**：必须演示跨会话记忆效果（第1天输入信息，第2天能使用）
- **主动服务**：必须演示主动推送效果（定时消息或主动提醒）

### 5.3 如实反映问题

- 遇到问题不丢分，隐瞒问题才丢分
- 问题描述应包含：问题现象、原因分析、解决过程
- 未解决的问题也要如实说明

### 5.4 文档中展示核心代码

- 在功能列表和技术实现部分，展示少量核心代码片段（每段不超过15行）
- 代码片段应体现关键技术点，如意图识别规则、记忆存储逻辑等
- 不需要提交完整代码仓库

### 5.5 必须总结与Agent协作经验

- 使用编程Agent（如CodeArts、Claude Code、Cursor等）辅助开发，并总结协同经验
- 重点不是"Agent帮你写了多少代码"，而是"你与Agent如何协同——你做决策和验证，Agent做实现和迭代"
- 至少描述1个典型协作案例，总结2-3条经验教训

## 六、课堂汇报要求

### 6.1 汇报形式

| 项目     | 要求               |
| ------ | ---------------- |
| **时长** | 每人3分钟，严格控时       |
| **形式** | PPT + 在线演示/演示视频  |
| **重点** | 做了什么 + 运行效果，紧凑快速 |

> **时间紧张**：必须紧凑高效，直奔重点。

### 6.2 汇报内容结构

| 部分     | 时间  | 内容                   |
| ------ | --- | -------------------- |
| 进展总览   | 30秒 | 一句话：完成度XX%，核心功能X个已完成 |
| 核心功能演示 | 2分钟 | 在线操作或播放视频，展示3个核心功能   |
| 问题与计划  | 30秒 | 最大问题+下一步重点           |

> **汇报技巧**：不要念PPT，直接演示效果。进展总览用一页带数字的幻灯片即可，把时间留给演示。

### 6.3 汇报评分要点

| 要点       | 说明                  |
| -------- | ------------------- |
| **有效演示** | 在线演示或视频播放，能看到真实运行效果 |
| **真实进展** | 如实汇报，不夸大不隐瞒         |
| **节奏紧凑** | 3-5分钟内完成，重点突出不拖沓    |

## 七、常见问题

### Q1：中期完成度多少算合格？

- **最低要求**：核心功能完成50%以上，长期记忆和主动服务至少实现一个
- **良好**：核心功能完成70%以上，两个核心特性都有实现
- **优秀**：核心功能完成80%以上，两个核心特性效果明显

### Q2：如果进度严重滞后怎么办？

- **如实汇报**：说明滞后原因和调整计划
- **展示已有成果**：即使功能不完整，也要展示已完成的部分
- **调整计划**：重新规划后续功能，可能需要削减P2功能
- **不要虚报**：虚报进度比进度滞后更严重

### Q3：核心功能怎么演示？

- **优先**：部署到公网，汇报时直接访问在线地址操作演示
- **备选**：公网部署条件不允许时，提前录制3分钟内演示短视频，汇报时播放
- 两种方式都需要在文档中提供地址或视频链接
- 不接受纯PPT文字描述，必须看到真实运行效果

### Q4：长期记忆怎么演示跨会话效果？

```
演示步骤：
1. 第1次会话：告诉助手"我今年55岁，膝盖不太好"
2. 助手回复：记录了信息
3. 关闭会话（刷新页面或重启服务器）
4. 第2次会话：问"推荐一些运动"
5. 助手回复中使用了之前的信息："根据您的年龄和膝盖情况..."
6. 截图两次会话，证明记忆跨会话有效
```

### Q5：与Agent协作总结怎么写？

不需要老师案例那样复杂，重点关注：

1. **协同方式**：你用了哪个Agent、怎么用的
2. **一个案例**：描述一次典型的人-Agent协作过程（你提需求→Agent实现→你发现问题→修复迭代）
3. **几条经验**：你觉得与Agent协同最重要的经验是什么

**写好写差的关键**：是不是体现"协同"而不是"代工"。说"Agent帮我写完了"是0分意识，说"我指定了功能位置和迭代节奏，Agent实现后我逐步验证发现了2个Bug并引导修复"是满分意识。
